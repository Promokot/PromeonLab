# Stage 5 Review ŌĆö Responsibility (SRP) & Duplicated Logic

**Date:** 2026-06-03
**Scope:** `Assets/_App/Scripts/` (runtime) + `Assets/_App/Editor/BuiltinRecipeBaker.cs`
**Mode:** Read-only analysis. All findings are *proposals* ŌĆö nothing was changed.

This stage hunts for two failure modes:

1. **SRP violations / god classes** ŌĆö one type holding several unrelated concerns, which makes the file hard to change safely and easy to break.
2. **Duplicated logic that must stay in sync** ŌĆö the same rule (sampling, transform math, ID/path parsing, collider setup, frameŌåötime conversion) coded in 2+ places. When one copy is edited and the other is not, you get silent visual/animation artifacts.

Severity legend: **High** = likely to cause bugs/artifacts now or actively blocks maintenance ┬Ę **Med** = real risk, currently contained ┬Ę **Low** = cleanliness / future-proofing.

## Top 5 to address first

1. **B3 ŌĆö unify the two sampling paths** (`AnimationAuthoring.cs:558-567` vs `:579-599`). Highest artifact risk: scrub-vs-play pose divergence at the same frame. Smallest, safest fix (one float-param method).
2. **B1 ŌĆö single diamond-mesh builder** (`RigEntityFactory.cs:217-279`). Duplicated vertex/triangle data for the *same* bone shape; trivial to extract, prevents inconsistent skeletons.
3. **B2 ŌĆö one transform-commit protocol** (`XRPromeonInteractable.cs:165-189` vs `GizmoActivator.cs:430-449`). Two undo conventions on one stack ŌåÆ wrong-baseline undo bugs.
4. **A1 ŌĆö break up `AnimationAuthoring`** into CRUD-fa├¦ade + `AnimationClipBaker` + `AnimationPlaybackSampler` + `AnimationStore`. Unblocks B3/B4 cleanly and is the biggest maintainability win.
5. **A2 ŌĆö extract `GizmoHighlightPainter` + `GizmoDragSession`** from `GizmoActivator`. Second-largest file; the highlight system is self-contained and historically bug-prone.

- `InteractionCapability.cs` is correctly the **single** "make this selectable" definition; builders produce only geometry and the registry calls `Apply` once (`AssetEntityBuilderRegistry.cs:35-42`). No duplication there.
