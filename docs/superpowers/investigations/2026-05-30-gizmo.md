# Gizmo investigation (2026-05-30)

READ-ONLY investigation. No project files were modified. Covers three gizmo work items:
3 (sizing model), 3.1 (grabbed-part highlight), 3.2 (global/local origin toggle button).
Companion to `docs/superpowers/investigations/2026-05-30-outline-see-through.md` — the
"gizmo visible through an occluding object" symptom is the QuickOutline see-through stencil
technique analysed there; THIS doc owns the part-highlight material swap (3.1).

## Summary

- **3 (Sizing).** The gizmo instance is uniformly scaled to fit the selected object's combined
  renderer bounds via `BoundsFitter.ComputeSize` (`BoundsFitter.cs:5-17`), called at spawn
  (`GizmoActivator.cs:133-134`) and re-fit after a scale commit (`GizmoActivator.cs:221-225`).
  The fit uses `maxExtent * boundsCoefficient` clamped to `[MinSize, MaxSize]` from `GizmoConfig`.
  To get a "roughly fixed size ±50%", replace the bounds-driven term with a single exposed
  `NormalSize` and clamp only to `[NormalSize*0.5, NormalSize*1.5]` — the clamp belongs in
  `GizmoConfig` (new fields) and `BoundsFitter` becomes a thin clamp helper (or is bypassed).
- **3.1 (Highlight).** The yellow material-swap is *already implemented* in
  `GizmoHierarchy.ApplyHighlight` (`GizmoHierarchy.cs:80-112`) and is wired on grab
  (`GizmoActivator.cs:166` → `GizmoHierarchy.OnHandleGrabbed` → `ApplyHighlight`). Root cause of
  "doesn't work": `ApplyHighlight` enumerates `handle.GetComponentsInChildren<Renderer>()`, but it
  also **reparents axis siblings into the grabbed handle immediately afterward**
  (`GizmoHierarchy.cs:61-70` `SetAsParent`) AND the highlight overwrites `sharedMaterials[0]`, which
  on these gizmo parts is **QuickOutline's appended outline material**, not the visible base mesh
  material (see "Material order" below). Net effect: the wrong material slot is tinted (or a
  QuickOutline pass is destroyed), so no visible yellow on the mesh. Fix = tint the correct base
  slot, scope to the handle's own renderer only (not children), and detect grab/release exactly
  where it is detected today.
- **3.2 (Origin toggle).** Buttons on `GizmoToolsPanel` publish `GizmoModeChangedEvent` over
  `EventBus`; `GizmoActivator` subscribes and drives the gizmo. There is currently **no origin/pivot
  concept** — the gizmo always sits at `_target.position` (the transform origin = "global"), set in
  `LateUpdate` (`GizmoActivator.cs:75-76`) and `Spawn` (`GizmoActivator.cs:127-128`). A 4th toggle
  button + a new `GizmoOriginModeChangedEvent` + a pivot-offset computed from the target's renderer
  bounds center gives the local/global switch.

## 3 — Sizing: bounding-box → fixed ±50%

### Current logic (evidence)

- `BoundsFitter.ComputeSize(Transform target, float boundsCoefficient, float minSize, float maxSize)`
  combines all child `Renderer.bounds`, takes the largest extent, and returns
  `Mathf.Clamp(maxExtent * boundsCoefficient, minSize, maxSize)` — `BoundsFitter.cs:5-17`.
- Spawn applies it as a **uniform** scale: `_instance.transform.localScale = Vector3.one * size`
  — `GizmoActivator.cs:133-134`.
- After a scale-drag commit it re-fits (bounds changed): `GizmoActivator.cs:221-225`.
- Config exposes the three knobs: `_boundsCoefficient` (Range 0.5–5, default 1.5), `_minSize`
  (0.01–1, default 0.1), `_maxSize` (1–20, default 5) — `GizmoConfig.cs:6-14`;
  asset values 1.5 / 0.1 / 5 in `DefaultGizmoConfig.asset:16-18`.

Consequence: a large imported model produces a huge gizmo (extent×1.5, up to MaxSize=5), a tiny
object a tiny gizmo — the exact "inconvenient" behavior described.

### Proposed model (fixed size ±50%)

Goal: gizmo size is essentially constant (a single tunable "normal size"), with at most ±50%
leeway, independent of the object's bounding box.

1. **Where the parameters live — `GizmoConfig`.** Add:
   - `[SerializeField] private float _normalSize = 0.4f;` (the "normal size"; the task notes it will
     be reduced later, so keep it a single exposed knob).
   - `[SerializeField, Range(0f, 0.5f)] private float _sizeLeeway = 0.5f;` (the ±fraction).
   Expose `NormalSize` / `SizeLeeway`. The clamp range is then
   `[NormalSize*(1-SizeLeeway), NormalSize*(1+SizeLeeway)]`. Keeping the leeway as a fraction makes
   "±50%" explicit and lets the normal size shrink later without re-tuning the clamp.
   - `_boundsCoefficient`/`_minSize`/`_maxSize` can be retained for back-compat but are no longer the
     primary driver; recommend deprecating `_boundsCoefficient` and repurposing min/max as the
     hard floor/ceiling if desired.

2. **Two viable size policies — pick one:**
   - **(A) Constant size (simplest, matches "fixed").** Always use `NormalSize`. The ±50% leeway is
     then a *clamp guard* only relevant if you keep any bounds influence. Net:
     `localScale = Vector3.one * NormalSize`.
   - **(B) Mild bounds influence, hard-clamped to ±50% (recommended).** Keep a *gentle* bounds term
     so the gizmo still grows slightly for very large objects, but never beyond ±50%:
     `size = Mathf.Clamp(maxExtent * boundsCoefficient, NormalSize*(1-leeway), NormalSize*(1+leeway))`.
     This is a one-line change to `BoundsFitter.ComputeSize`'s clamp bounds and preserves its call
     sites unchanged.

3. **Where to change call sites.** Both spawn (`GizmoActivator.cs:133`) and post-commit refit
   (`GizmoActivator.cs:223`) call `BoundsFitter.ComputeSize(...)`. If you adopt (B), change only the
   two clamp arguments passed there (or the `BoundsFitter` signature) — the rest of the activator is
   untouched. If you adopt (A), `BoundsFitter` is no longer needed for sizing and both lines become
   `_instance.transform.localScale = Vector3.one * _config.NormalSize;`.

4. **Important interaction with scale-drag inversion.** During a scale drag the gizmo mutates its own
   `localScale` and the target follows via the ratio `instance.scale / instanceScaleAtGrab`
   (`GizmoActivator.cs:187-199`, baselines captured at `:163-164`). The fixed-size model does **not**
   break this — the baselines are captured at grab from whatever the current instance scale is, and
   the post-release refit (`:221-225`) re-applies the fixed/clamped size. Keep the refit call; just
   feed it the new clamp.

Recommendation: **(B)** — minimal diff, keeps `BoundsFitter` and both call sites, and the ±50% clamp
is enforced in exactly one place (`BoundsFitter.cs:16`).

## 3.1 — Grabbed-part highlight (yellow material swap)

### How parts are identified on grab (evidence)

- Each axis/center/ring is a `GizmoHandle` (`GizmoHandle.cs`), an `XRBaseInteractable` carrying a
  `HandleKind` (`MoveAxis`/`RotateRing`/`ScaleAxis`/`ScaleUniform`, `HandleKind.cs`) and an
  `AxisKind` (`X`/`Y`/`Z`, `AxisKind.cs`). Selection-by-interactor is disabled
  (`IsSelectableBy => false`, `GizmoHandle.cs:41`); the handle drives grab itself via grip input in
  `ProcessInteractable` (`GizmoHandle.cs:43-111`).
- Grab is detected in `GizmoHandle.cs:76-89` ("GRIP DOWN — entering Dragging") → calls
  `_activator.OnHandleGrabbed(this, virtualPos, ctrl.rotation)`. Release is `GizmoHandle.cs:97-104`
  ("GRIP UP") → `_activator.OnHandleReleased()`. A defensive abort path exists at
  `GizmoHandle.cs:48-55`.
- `GizmoActivator.OnHandleGrabbed` forwards to the hierarchy: `_hierarchy?.OnHandleGrabbed(handle)`
  (`GizmoActivator.cs:166`). Release/abort call `_hierarchy.OnHandleReleased(_mode)`
  (`GizmoActivator.cs:210, 245`).
- In the prefab, each handle GameObject is an **instance of an FBX model prefab** (Move/Rotate/Scale
  FBX, source guids `80ea28f1…`, `e7c4cc77…`, `5a933a6e…`) with a `GizmoHandle`, a `MeshCollider`
  for hit-testing, **and a QuickOutline `Outline` component** all on the same GameObject
  (`Vr3D_Gizmos.prefab` — `GizmoHandle` at lines 267, 386, 505, 731…; `QuickOutline::Outline` at
  168, 227…; `MeshCollider` at 237, 356…). The visible `MeshRenderer` is inherited from the FBX
  prefab instance (not serialized as an explicit block, which is why a raw grep shows
  `MeshRenderer: 0`).

### Highlight is already attempted — and why it fails (root cause)

The yellow swap exists today in `GizmoHierarchy`:

- On grab, `OnHandleGrabbed` calls `ApplyHighlight(handle)` *before* reparenting
  (`GizmoHierarchy.cs:55-71`).
- `ApplyHighlight` (`GizmoHierarchy.cs:80-112`): for every `Renderer` under the handle it clones
  `sharedMaterials[0]`, tints it yellow (`_highlightColor`, default `(1, 0.85, 0.1)` at
  `GizmoHierarchy.cs:27`) via `_BaseColor`/`_Color`/`_EmissionColor`, then writes a new array where
  slot 0 = tinted clone and slots `1..n` keep the originals (`:100-103`). On release/abort
  `ClearHighlight` restores `OriginalSharedMaterials` and destroys the clone (`:114-122`).

Why no visible yellow:

1. **Material-slot collision with QuickOutline (primary suspect).** QuickOutline's `Outline.OnEnable`
   **appends** two material instances (OutlineMask, OutlineFill) to each renderer's material list
   (documented in the companion doc, `2026-05-30-outline-see-through.md`, "How QuickOutline
   renders…", citing `Outline.cs:102-113`). So at runtime the live `sharedMaterials` array is
   reordered/owned by QuickOutline. `ApplyHighlight` assumes slot 0 is the base mesh material, but
   after QuickOutline runs, slot 0 may be an outline pass (or QuickOutline rebuilds the array on its
   own `Update`, discarding the tinted clone). Tinting/replacing slot 0 then either colors an
   invisible mask pass or is silently reverted by QuickOutline the next frame. Net: base mesh color
   never changes.
2. **Reparent timing widens the renderer set.** `OnHandleGrabbed` calls `ApplyHighlight` then
   `SetAsParent(handle.transform, …siblings)` (`GizmoHierarchy.cs:61-67`). The comment at
   `:58-59` says highlight is taken before reparenting *so siblings don't get tinted* — correct in
   intent, but it confirms `GetComponentsInChildren<Renderer>` is the enumeration basis, which is
   fragile: if a handle's mesh is actually on a child (FBX sub-mesh) the array length/order differs
   from the assumption that slot 0 is the base material.
3. **Shared vs instanced material confusion.** Writing `r.sharedMaterials` mutates the *shared*
   asset binding for that renderer. If multiple handles share one material asset (they likely do —
   all X-axis parts reference `Gizmo_Red.mat`, etc. under `Content/Materials/Gizmo/`), the restore
   path is still per-renderer so it's safe, but the clone-on-slot-0 approach is sensitive to which
   slot QuickOutline considers "the base".

### Proposed material-swap design

Keep the existing structure (`ApplyHighlight`/`ClearHighlight` + grab/release wiring is correct);
make the swap robust to QuickOutline and to mesh-on-child layouts:

1. **Pre-create a yellow variant per base material (don't clone slot 0 blindly).** The task says
   "swap to the SAME material but yellow." There are exactly 3 colored gizmo base materials
   (`Gizmo_Red.mat`, `Gizmo_Green.mat`, `Gizmo_Blue.mat`) plus `Gizmo_Default.mat` under
   `Assets/_App/Content/Materials/Gizmo/`. Either (a) author one `Gizmo_Yellow.mat` and swap the
   *base* slot to it, or (b) keep cloning but identify the base slot by **excluding QuickOutline
   passes** (skip materials whose shader name contains `Outline` / `OutlineMask` / `OutlineFill`)
   rather than assuming index 0.
2. **Identify the base slot explicitly.** In `ApplyHighlight`, instead of `originals[0]`, find the
   first material whose `shader.name` is NOT a QuickOutline outline shader; tint/replace that slot,
   preserve all others (including the outline passes) in place. Store
   `{Renderer, originalSharedMaterials[], replacedIndex, tintedClone}` so `ClearHighlight` restores
   exactly.
3. **Scope to the handle's own renderer.** Use the renderer on `handle` (or its direct mesh child),
   not a broad `GetComponentsInChildren` after reparenting, to avoid tinting siblings that were just
   re-parented in. Take the highlight snapshot strictly before `SetAsParent` (already the order
   today — preserve it).
4. **Where original material is stored / yellow variant / detect grab-release:**
   - *Original* → in the existing `HighlightedRenderer.OriginalSharedMaterials`
     (`GizmoHierarchy.cs:39-44`) — keep.
   - *Yellow* → either a serialized `[SerializeField] Material _highlightMaterial` on `GizmoHierarchy`
     (cleanest, no per-frame allocation, and matches "same material but yellow"), or the existing
     runtime clone but tinting the correctly-identified base slot.
   - *Detect grab* → already at `GizmoHandle.cs:76-89` → `OnHandleGrabbed`; *detect release* → already
     at `GizmoHandle.cs:97-104` (and abort at `:48-55`) → `OnHandleReleased`. No new detection
     needed; only fix what gets tinted.
5. **Cross-reference (NOT in scope here).** The separate concern "gizmo visible *through* an
   occluding object" is the QuickOutline see-through stencil/ZTest behavior owned by
   `2026-05-30-outline-see-through.md`. The yellow base-material swap (this item) is independent: it
   recolors the solid mesh, not the outline. If a `Gizmo_Yellow.mat` is authored, ensure its ZTest
   matches the other gizmo materials so the highlight doesn't itself become see-through.

## 3.2 — Global/local origin toggle button

### Current button wiring (evidence)

- `GizmoToolsPanel` (`GizmoToolsPanel.cs`) holds three serialized `Button`s — `_moveButton`,
  `_rotateButton`, `_scaleButton` (`:7-9`) — plus optional active-indicator GameObjects (`:11-13`).
- `Awake` adds click listeners that call `SelectMode(GizmoMode.X)` (`:21-26`); `SelectMode` sets
  `_current`, updates indicators, and **publishes `GizmoModeChangedEvent { Mode }`** over the
  injected `EventBus` (`:52-57`).
- On enable it publishes `GizmoToolsPanelOpenedEvent` and subscribes to drag start/end to
  enable/disable the buttons (`:28-50, 66-74`); on disable it publishes `GizmoToolsPanelClosedEvent`.
- `GizmoActivator` subscribes to all of these in `Construct` (`GizmoActivator.cs:43-46`) and reacts:
  `OnModeChanged` sets `_mode` and calls `_hierarchy.ShowMode` (`:94-99`); panel open/close drive
  `RefreshVisibility`/spawn-despawn.
- Prefab wiring: in `GizmoToolsModule.prefab` the `GizmoToolsPanel` MonoBehaviour
  (`m_Script` guid `8efef404…`, lines 230-247) has `_moveButton`/`_rotateButton`/`_scaleButton`
  pointing at the three `Button` components (fileIDs `9172621615216711491`, `2561343058307554984`,
  `1542034994893271325`). The three button GameObjects (`GizmoMoveBtn`/`GizmoRotateBtn`/
  `GizmoScaleBtn`) live under a `HorizontalLayoutGroup` GameObject "Buttons Raw"
  (`m_Children` at lines 957-960, layout at `:968-993`, panel size 400×90 / row 470×100). Each button
  = RectTransform + UI.Button + Image + child ImageIcon + child Label (TMP).

### How the gizmo currently decides its origin/pivot (evidence)

There is **no pivot concept today** — the gizmo is always pinned to the target's transform origin
("global"):

- `Spawn` sets `_instance.transform.position = _target.position` (`GizmoActivator.cs:127`).
- `LateUpdate` keeps it there each frame when not dragging:
  `_instance.transform.position = _target.position; …rotation = _target.rotation`
  (`GizmoActivator.cs:75-76`).
- Every drag strategy also pivots about the target's transform origin/position, not a bounds center:
  `AxisMoveStrategy` uses `target.position` (`AxisMoveStrategy.cs:14`), `RingRotateStrategy` uses
  `_targetPosAtGrab = target.position` (`RingRotateStrategy.cs:17`), both scale strategies use
  `target.position` as the reference (`AxisScaleStrategy.cs:20`, `UniformScaleStrategy.cs:16`).
- `BoundsFitter` already computes a combined renderer `Bounds` (`BoundsFitter.cs:10-12`) — its
  `.center` is exactly the **local bounding-box center** needed for the "local" origin mode (it is
  computed there but currently only `.extents` is used).

### Proposed toggle design

1. **State location.** The origin mode is gizmo-presentation state, so own it in `GizmoActivator`
   (parallel to `_mode`): add `private GizmoOriginMode _originMode = GizmoOriginMode.Global;` and a
   new enum file `GizmoOriginMode { Global, Local }` (one public type per file, in
   `Scripts/VrInteraction/`). The panel only holds transient UI `_currentOrigin` for the indicator,
   exactly like `_current` for mode.
2. **New event (cross-subsystem rule).** Add `GizmoOriginModeChangedEvent { public GizmoOriginMode Mode; }`
   in `Scripts/VrInteraction/Events/` (mirrors `GizmoModeChangedEvent.cs:1`). The panel publishes it;
   `GizmoActivator` subscribes in `Construct` and unsubscribes in `OnDestroy` (mirror
   `GizmoActivator.cs:43-46, 53-57`). No direct method calls across the boundary.
3. **Compute the pivot per mode in `GizmoActivator`.** Add a helper:
   - `Global` → `_target.position` (current behavior, no offset).
   - `Local` → the world-space bounds center: compute the combined renderer bounds (reuse/extend
     `BoundsFitter` to also return `bounds.center`, since it already builds the bounds at
     `BoundsFitter.cs:10-12`) and place the gizmo there. Concretely
     `_instance.transform.position = boundsCenterWorld` instead of `_target.position`.
   - Apply this in the two places the position is set: `Spawn` (`GizmoActivator.cs:127`) and the
     non-drag `LateUpdate` (`:75`). On `OnOriginModeChanged`, if not dragging, re-place the instance
     (cheap; or `Despawn()/Spawn()` to reuse the existing refresh path at `:116`).
   - **Drag note:** the strategies pivot about `target.position`. For a faithful "local" pivot you
     would also pass the chosen pivot into `BeginDrag` (the `IGizmoDragStrategy` signature at
     `IGizmoDragStrategy.cs:5` currently takes `handPos` but not a pivot). Minimum viable: only move
     the gizmo's *visual* origin for now (position the instance at the bounds center) and leave drag
     math pivoting on `target.position`; full local-pivot rotation/scale is a follow-up that threads
     a pivot Vector3 through the strategies. Flag this in Open Questions.
4. **Add the 4th button (consistent with the others).**
   - *Code* — in `GizmoToolsPanel`: add `[SerializeField] private Button _originButton;` and optional
     `_originActiveIndicator`; in `Awake` add
     `_originButton.onClick.AddListener(ToggleOrigin)` (mirrors `:23-25`); add
     `ToggleOrigin()` that flips `_currentOrigin` and publishes `GizmoOriginModeChangedEvent`
     (mirror `SelectMode`, `:52-57`); include it in `SetButtonsInteractable` (`:69-74`) and the
     indicator update (`:59-64`). It is a *toggle*, not part of the mutually-exclusive mode set, so
     keep it separate from `_current`.
   - *Prefab* — duplicate one of the existing button GameObjects (e.g. `GizmoScaleBtn`,
     fileID `5615339259401905641`) under "Buttons Raw" (`Vr3D`… no — `GizmoToolsModule.prefab`
     RectTransform `2028684543701688472`, children list at lines 957-960), set its Label to
     "Local"/"Global", and assign the new `Button` to `GizmoToolsPanel._originButton`. The
     `HorizontalLayoutGroup` (`:968-993`) auto-arranges it; widen panel `m_SizeDelta` (`:166`, row
     `:966`) if four buttons need more room.

## Verification steps

1. **Sizing (3).** In play mode, select objects of very different sizes (a large imported model and a
   small primitive). Confirm the gizmo's on-screen size is now near-constant — between
   `NormalSize*0.5` and `NormalSize*1.5` — regardless of object bounds. Verify a scale-drag still
   works and the post-release refit (`GizmoActivator.cs:221-225`) re-clamps to the new range.
2. **Highlight (3.1).** Grab each handle (X/Y/Z move axes, the move center, each rotate ring, scale
   axes, uniform scale). Confirm the grabbed part's *solid mesh* turns yellow and reverts on release
   and on abort (`GizmoHandle.cs:48-55`). Verify the QuickOutline silhouette still renders (the swap
   must not destroy outline passes). Check the Console: the existing
   `[GizmoActivator] OnHandleGrabbed` log (`GizmoActivator.cs:157`) confirms the grab path fires; if
   it fires but no yellow appears, the slot-identification fix is the lever.
3. **Origin toggle (3.2).** With an off-center mesh (transform origin ≠ bounds center), toggle the
   4th button: in **Global** the gizmo sits at the transform origin (current behavior); in **Local**
   it jumps to the renderer bounds center. Confirm `GizmoOriginModeChangedEvent` is published once
   per click and that toggling mid-selection re-places the gizmo without needing a re-select. Confirm
   the button disables during drag like the others (`GizmoToolsPanel.cs:69-74`).

## Open questions

1. **Sizing policy A vs B.** Constant size (A) vs gently bounds-influenced + ±50% clamp (B)? B is
   recommended but needs a `_normalSize` value chosen against the (soon-to-be-reduced) gizmo art.
2. **QuickOutline ownership of the material array.** Does QuickOutline rebuild `sharedMaterials` on
   its `Update` after our swap (which would silently revert the tint)? If so, the yellow swap must be
   re-applied after QuickOutline, or applied via `MaterialPropertyBlock` on the base slot instead of
   array replacement. Confirm by inspecting whether `ClearHighlight` ever runs with a stale array.
   This is the single biggest risk to 3.1 and overlaps the companion outline doc.
3. **Base-slot identification.** Are the gizmo mesh renderers single-material, or do the FBX sub-meshes
   carry multiple slots? Determines whether "skip outline shaders, take first remaining" is
   sufficient or whether an explicit serialized `Material _highlightMaterial` is safer.
4. **Local pivot during drag.** Should move/rotate/scale actually pivot about the bounds center in
   Local mode, or is moving only the gizmo's visual origin enough for v1? Full local-pivot requires
   threading a pivot `Vector3` through `IGizmoDragStrategy.BeginDrag` (`IGizmoDragStrategy.cs:5`) and
   all four strategies.
5. **Toggle button label/icon.** Should the 4th button show a single toggling label ("Local"↔"Global")
   or two icons? The other buttons use ImageIcon + Label; reuse that pattern.
6. **`TransformGizmoHierarchyController` (SceneComposition).** The gizmo prefab root also carries a
   `Subsystems.SceneComposition::TransformGizmoHierarchyController` (`Vr3D_Gizmos.prefab:47-49`,
   guid `98ac50b5…`) — not in `Scripts/VrInteraction/Gizmo/` and not referenced by the activator path
   reviewed here. Confirm it is dead/legacy before changing prefab structure for 3.2.
