# Варианты правок 3.2.1: «каркас» как условная группировка

Статус: НА ВЫБОР Максом (2026-06-04). После выбора файл удаляется, решения уходят в plan-3.2.md.
Контекст: ревью Макса – не наделять каркас задачами/ролями (это условная группировка механизмов) + замена термина «сервис» → «объект»/«класс» (решено отдельно).

## Место 1 – первый абзац пункта

СЕЙЧАС:
> Каркас приложения решает три задачи: связывает модули между собой, управляет временем жизни сервисов и переключает режимы работы. Для связи модулей используется обмен сообщениями через общую шину, для управления временем жизни – архитектура внедрения зависимостей (Dependency Injection, DI), для переключения режимов – оркестратор с графом разрешённых переходов. Эти механизмы рассматриваются по порядку, однако сначала следует обозначить основание, на котором стоит весь каркас, – технологии виртуальной реальности.

ВАРИАНТ А (компактный – одно определяющее предложение):
> Архитектурным каркасом приложения далее называется условная группа из трёх механизмов, общих для всех модулей: обмена сообщениями через общую шину, управления временем жизни объектов средствами внедрения зависимостей (Dependency Injection, DI) и переключения режимов работы по графу разрешённых переходов. Эти механизмы рассматриваются по порядку, однако сначала следует обозначить общее для них основание – технологии виртуальной реальности.

ВАРИАНТ Б (два предложения, ближе к нынешней структуре):
> Архитектурным каркасом приложения далее называется условная группа механизмов, общих для всех модулей. К ней отнесены обмен сообщениями через общую шину, управление временем жизни объектов на основе внедрения зависимостей (Dependency Injection, DI) и переключение режимов работы через оркестратор с графом разрешённых переходов. Эти механизмы рассматриваются по порядку, однако сначала следует обозначить общее для них основание – технологии виртуальной реальности.
