## Проверка по коду (сводка фактов)

- МИНИ-ИССЛЕДОВАНИЕ (решение 10, подтверждено архивом спек docs/superpowers/_archive/specs/2026-05-20-*): первый подход строился на пакете Unity Animation Rigging – визуализация костей пакета (BoneRenderer) работает только в редакторе и в сборке не отображается (отсюда собственные меши-ромбы), а привязка костей к прокси делалась констрейнтом MultiParentConstraint пакета. Подход отклонён («v2-proxy-skeleton»: сложная настройка Rig/RigBuilder, ненадёжность при VR-манипуляции) – констрейнты заменены СОБСТВЕННЫМ компонентом BoneFollower (копирование локальных трансформаций в LateUpdate), зависимость из кода удалена полностью. Подача: обоснование выбора решения, факт «опробовано и заменено», НЕ как готовая функциональность.
- Описание скелета в шаблоне: RigDefinition – версия схемы, идентификатор, ось концевых костей с инверсией (значения из мастера импорта, см. 3.2.4), список BoneRecord (имя кости + признак блокировки перемещения). Поле IkChains потребителя не имеет (солвер не реализован) – в листинге опускается, в тексте не упоминается (решение 3).
- Извлечение при импорте: RigDefinitionExtractor.FromSkinnedMesh (19 строк, чистая функция, объектов не строит): кости SkinnedMeshRenderer → записи BoneRecord; нет пригодного скелета → null → модель деградирует до статического объекта (в 3.2.4 об этом уже сказано – не повторяться).
- InteractionCapability.Apply – только связующая фраза (решение 5): вид коллайдера BoneBoxes в шаблоне означает, что коллайдеры строит фабрика рига; без листинга, без Б.
- Построение прокси-рига: RigEntityFabricator.BuildProxyRig (вызов из RigEntityBuilder.RestoreAsync, для импортных и встроенных): поиск живых костей по именам шаблона (ResolveTransforms; пустой список имён = все кости скинированного меша – случай встроенных), корень ProxyRig повторяет локальные трансформации арматуры, рекурсивный BuildProxyNode по выбранным костям. Вместе с прокси фабрика строит селекторные боксы выбора всего рига по скелету до глубины из шаблона – пара предложений, без деталей планировщика и без листинга (решение 6).
- Геометрия прокси-кости: общий 6-вершинный ромб; кость с детьми – комбинированный меш по направлениям к каждому ребёнку; концевая кость – направление по оси из шаблона (Auto = по направлению от родителя), длина = половина смещения от родителя, ширина ограничена долей длины. Связка с переключателем оси в мастере (3.2.4).
- Состав прокси-объекта (BuildProxyNode): меш-ромб + рендер, обводка QuickOutline, выпуклый меш-коллайдер, компоненты SceneNode + маркер BoneSceneNodeMarker + Selectable + интерактабл слоя BoneProxies; на настоящую кость вешается BoneFollower со ссылкой на прокси (со снятием устаревших).
- Связь прокси → кость: BoneFollower (37 строк) – каждый LateUpdate копирует локальные позицию и поворот прокси на кость; масштаб не копируется, а умножается на захваченный базовый масштаб кости (сохраняет неединичный масштаб покоя; растяжение детей – через иерархию localScale).
- Владение и состояние: ProxyRigRuntime (200 строк, по одному на риг, Bind от фабрики): хранит прокси, селекторные коллайдеры, словарь «имя кости → прокси»; позы – CapturePoses/ApplyPoses по локальным трансформациям прокси (структура BonePose – 3.1.2; применение при загрузке – 3.2.3; отсылки минимальные по 5.13); механизм скрытия/показа костей SetBonesInteractive – боксы и кости включаются взаимоисключающе; стартовое состояние – выбор всего рига. Упоминаем НАЛИЧИЕ механизма без отсылок куда-либо (решение 7); сценарий использования – не здесь.
- Идентификаторы костей в графе (кратко, решение 2/8): SceneGraph по маркерам переписывает идентификаторы прокси в форму bone:{rigNodeId}:{boneName} и регистрирует их транзитными нодами – находимы по идентификатору, невидимы обозревателю. Без механики выбора.
- ⚠ Без IK и «ограничителей»; BoneEditMode, выбор и подсветка костей, реакция на SelectionChangedEvent – тема 3.2.6 (в тексте 3.2.5 не разбираются).

## Структура текста (порядок абзацев)

1. Вводный: кости скинированного меша – внутренние трансформации без собственной геометрии, напрямую в VR их не выбрать и не схватить. Подхват финала 3.2.4.
1а. Мини-исследование (выбор решения): готовый пакет Unity Animation Rigging – отображение костей редакторское, в сборке недоступно; его констрейнт-механизм привязки опробован и отклонён (сложность настройки, ненадёжность при манипуляции в VR) → собственное решение: прокси-кости с видимой геометрией + компонент следования BoneFollower. Возможен маркер [ИСТ: Animation Rigging].
2. Описание скелета: RigDefinition и BoneRecord [Листинг 3.40, без IkChains], извлечение при импорте [Листинг 3.41 RigDefinitionExtractor целиком].
3. Построение прокси-рига: точка вызова при восстановлении (связующая фраза про BoneBoxes из шаблона), поиск костей по именам, корень ProxyRig, рекурсия [Листинг 3.42 – фрагмент BuildProxyRig]; здесь же пара предложений о селекторных боксах всего рига.
4. Геометрия прокси-кости: ромбы, комбинированный меш, концевая кость и ось из мастера [Листинг 3.43 – фрагмент геометрии концевой кости].
5. Состав прокси-объекта: навешиваемые компоненты [Листинг 3.44 – фрагмент BuildProxyNode].
6. Связь прокси и кости: BoneFollower [Листинг 3.45 целиком], масштаб-множитель.
7. Владение ригом: ProxyRigRuntime – привязка, позы CapturePoses/ApplyPoses [Листинг 3.46 – фрагмент]; механизм скрытия/показа костей [Листинг 3.47 – фрагмент SetBonesInteractive], наличие без отсылок.
8. Кратко: регистрация прокси в графе сцены – маркер, идентификаторы bone:{rigNodeId}:{boneName}, транзитные ноды (невидимы обозревателю).
9. Финал-переход: построенные кости – объекты выбора и манипуляции, тема следующего пункта (словесно, правило 5.9). [СКРИНШОТ прокси-костей; СХЕМА прокси-рига]
