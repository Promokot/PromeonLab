# План пункта 3.2.5 «Построение скелетных структур»

Статус: УТВЕРЖДЁН с решениями (2026-06-05), готов к написанию текста.
База: plan-3.2.md (состав утверждён), код Assets/_App/Scripts/RigBuilder/ + AssetBrowser/RigEntityFabricator.cs, RigEntityBuilder.cs + VrInteraction/InteractionCapability.cs.
Свободные номера: Листинг 3.40+, Рисунок 3.15+, Таблица 3.4+, Б.20+.

## Проверка по коду (сводка фактов)

- Описание скелета в шаблоне: RigDefinition – версия схемы, идентификатор, ось концевых костей с инверсией (значения из мастера импорта, см. 3.2.4), список BoneRecord (имя кости + признак блокировки перемещения). Поле IkChainRecord потребителя не имеет (солвер не реализован) – по решению 3 в листинге опускается, в тексте не упоминается.
- Извлечение при импорте: RigDefinitionExtractor.FromSkinnedMesh (19 строк, чистая функция, объектов не строит): кости SkinnedMeshRenderer → записи BoneRecord; нет пригодного скелета → null → модель деградирует до статического объекта (в 3.2.4 одно предложение об этом уже есть – не повторяться, сослаться по сути).
- Оснащение восстановленного объекта по шаблону (РЕШЕНИЕ 4 – централизованный блок): InteractionCapability.Apply – единственная точка превращения геометрии в полноценный объект сцены. Идемпотентность: готовый интерактабл (испечённый встроенный префаб) пропускается. Порядок: 1) идентичность – компонент SceneNode (идентификатор проставит граф при регистрации); 2) коллайдер по виду из шаблона: Box – один BoxCollider на корне с центром и габаритами из шаблона; ConvexMesh – выпуклый MeshCollider на каждом отрисовываемом меше, нечитаемый меш (GPU-only после glTFast) пропускается с предупреждением, не прерывая построение; BoneBoxes – здесь не строится, боксы создаёт фабрика рига; 3) признак выбираемости – без него объект остаётся декорацией; 4) компоненты Selectable + XRPromeonInteractable, регистрация дочерних коллайдеров (корневой Awake-скан их не находит), назначение слоя взаимодействия одним вызовом. Слои коллайдерам Box/ConvexMesh назначаются ещё до проверки выбираемости.
- Построение прокси-рига при восстановлении: RigEntityFabricator.BuildProxyRig (вызов из RigEntityBuilder.RestoreAsync, для импортных и встроенных): поиск живых костей по именам шаблона (ResolveTransforms; пустой список имён = все кости скинированного меша – случай встроенных), корень ProxyRig повторяет локальные трансформации арматуры, рекурсивный BuildProxyNode – на каждую кость прокси-объект: меш-ромб, материал и обводка (QuickOutline), выпуклый меш-коллайдер, SceneNode + маркер BoneSceneNodeMarker, интерактабл слоя BoneProxies; на настоящую кость вешается BoneFollower со ссылкой на прокси.
- Геометрия прокси-кости: общий 6-вершинный ромб; кость с детьми – комбинированный меш по направлениям к каждому ребёнку; концевая кость – направление по оси из шаблона (Auto = по направлению от родителя), длина = половина смещения от родителя, ширина ограничена долей длины. Подробно словами, без отдельного листинга геометрии (решение 4: не выбран).
- Связь прокси → кость: BoneFollower (37 строк) – каждый LateUpdate копирует локальные позицию и поворот прокси на кость; масштаб не копируется, а умножается на захваченный базовый масштаб кости (сохраняет неединичный масштаб покоя; растяжение детей – через иерархию localScale).
- Владение и состояние: ProxyRigRuntime (200 строк, по одному на риг, Bind от фабрики): хранит прокси, селекторные коллайдеры, словарь «имя кости → прокси»; режимы – выбор всего рига (боксы включены, кости скрыты) / режим костей; позы – CapturePoses/ApplyPoses по локальным трансформациям прокси (структура BonePose – Листинг 3.4 в 3.1.2; применение при загрузке сцены – 3.2.3; минимум обратных отсылок по 5.13); RegisterSelectorColliders – после InteractionCapability.Apply, порядок описан в 3.2.4 одной фразой; реакция на SelectionChangedEvent (подсветка кости) – кратко, механика выбора в 3.2.6.
- Селекторные боксы всего рига: BoneSelectorBoxPlanner (66 строк, чистый планировщик) – обход скелета до глубины boneColliderDepth из шаблона (4), до предела бокс охватывает кость и прямых детей, на пределе – весь остаток поддерева; фабрика строит по плану BoxCollider в локальном AABB точек с минимальной толщиной (ширина кости × множитель из ProxyRigConfig – хитбоксы толще видимых костей). Словами, без листинга (решение 4: не выбран).
- ProxyRigConfig (ScriptableObject): материалы кости и выбранной кости, ширина, множитель толщины селекторов, вид коллайдера прокси.
- Идентификаторы костей в графе (РЕШЕНИЕ 2 – кратко здесь): SceneGraph по маркерам BoneSceneNodeMarker переписывает идентификаторы прокси в форму bone:{rigNodeId}:{boneName} и регистрирует их транзитными нодами – находимы по идентификатору, невидимы обозревателю. Использование при выборе кости – в 3.2.6.
- ⚠ Без IK-солвера и «ограничителей»; BoneEditMode (видимость костей, выбор) – тема 3.2.6.

## Структура текста (порядок абзацев) – «жизненный цикл скелета»

1. Вводный: скелетная модель управляется костями, но кости скинированного меша – внутренние трансформации без собственной геометрии, напрямую в VR их не выбрать и не схватить → поверх модели строится структура прокси-костей. Подхват перехода из финала 3.2.4.
2. Описание скелета: RigDefinition и BoneRecord [Листинг 3.40, поля без IkChains], извлечение при импорте RigDefinitionExtractor [Листинг 3.41 целиком]; деградация без скелета – по сути, без повтора 3.2.4.
3. ЦЕНТРАЛИЗОВАННЫЙ БЛОК: оснащение восстановленного объекта по шаблону – InteractionCapability.Apply: идемпотентность, порядок (идентичность → коллайдер по виду → выбираемость → интерактабл + слой), стратегии коллайдеров Box/ConvexMesh/BoneBoxes, устойчивость к нечитаемым мешам [Листинг 3.42 – фрагмент Apply; полный – Б.22]. Подача: общая операция для всех типов ассетов, у скелетных моделей вид BoneBoxes – коллайдеры строит фабрика рига (мост к следующему абзацу).
4. Построение прокси-рига: точка вызова (восстановление билдером – связка с 3.2.4 одной фразой), поиск костей по именам, корень ProxyRig, рекурсия по костям, состав прокси-объекта [Листинг 3.43 – фрагмент BuildProxyRig].
5. Геометрия прокси-кости словами: ромбы, комбинированный меш, концевая кость и ось из мастера (связка с переключателем оси в 3.2.4).
6. Связь прокси и кости: BoneFollower [Листинг 3.44 целиком], масштаб-множитель.
7. Владение ригом: ProxyRigRuntime – привязка, два режима (боксы/кости, кратко), позы CapturePoses/ApplyPoses [Листинг 3.45 – фрагмент поз] (отсылки 3.1.2/3.2.3 – минимум).
8. Селекторные боксы словами: планировщик, глубина, минимальная толщина; регистрация после Apply (одной фразой, без повтора 3.2.4).
9. Кратко: регистрация прокси в графе сцены – маркер, идентификаторы bone:{rigNodeId}:{boneName}, транзитные ноды (невидимы обозревателю); использование при выборе кости – словесная отсылка вперёд.
10. Финал-переход: построенные кости – объекты выбора и манипуляции, тема следующего пункта (словесно, правило 5.9). [СКРИНШОТ прокси-костей; СХЕМА прокси-рига]
