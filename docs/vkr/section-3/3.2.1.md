# 3.2.1 Архитектурный каркас приложения

> Статус: черновик v2 на ревью (расширение: XR-стек, точки входа, запуск, Runner/Bootstrap листинги, блок трёх режимов, итог).
> Нумерация сквозная в пределах раздела 3, реестр: numbering-registry.md.

Каркас приложения решает три задачи: связывает модули между собой, управляет временем жизни сервисов и переключает режимы работы. Для связи модулей используется обмен сообщениями через общую шину, для управления временем жизни – архитектура внедрения зависимостей (Dependency Injection, DI), для переключения режимов – оркестратор с графом разрешённых переходов. Эти механизмы рассматриваются по порядку, однако сначала следует обозначить основание, на котором стоит весь каркас, – технологии виртуальной реальности.

Работу с виртуальной реальностью обеспечивает рантайм OpenXR – открытый стандарт, через который приложение взаимодействует с гарнитурой, не привязываясь к программному обеспечению одного производителя [ИСТ: OpenXR]. Пользователя в сцене представляет объект XR Origin (XR-риг): он отслеживает положение головы и контроллеров и содержит камеру. Слой ввода построен на пакете XR Interaction Toolkit (XRI) [ИСТ: XRI]: поверх него работают и лучевая указка интерфейса, и взаимодействие с объектами сцены, рассматриваемые в дальнейших пунктах. Весь пользовательский ввод – от нажатия кнопки интерфейса до захвата объекта – проходит через эти механизмы.

Шину сообщений реализует класс EventBus. Сообщением служит типизированная структура с суффиксом Event в имени – такие сообщения уже встречались при описании автосохранения: SceneModifiedEvent, SceneOpenedEvent. Модуль-отправитель публикует структуру методом Publish, модули-получатели заранее оформляют подписку методом Subscribe; отправитель не знает своих получателей, и наоборот [ИСТ: паттерн Publish-Subscribe]. Прямые вызовы между модулями тем самым исключаются: добавление нового получателя сообщения не требует правок в отправителе. Полный текст класса приведён в Листинге 3.11.

Листинг 3.11 – Скрипт EventBus.cs
```csharp
public class EventBus
{
    private readonly Dictionary<Type, List<object>> _handlers = new();

    public void Subscribe<T>(Action<T> handler) where T : struct
    {
        var type = typeof(T);
        if (!_handlers.ContainsKey(type))
            _handlers[type] = new List<object>();
        _handlers[type].Add(handler);
    }

    public void Unsubscribe<T>(Action<T> handler) where T : struct
    {
        if (_handlers.TryGetValue(typeof(T), out var list))
            list.Remove(handler);
    }

    public void Publish<T>(T message) where T : struct
    {
        if (!_handlers.TryGetValue(typeof(T), out var list)) return;
        foreach (var handler in list.ToArray())
            ((Action<T>)handler).Invoke(message);
    }
}
```

Реализация компактна: словарь сопоставляет тип сообщения списку обработчиков. Существенная деталь скрыта в методе Publish: перед обходом список обработчиков копируется в массив, поэтому обработчик может в ответ на сообщение оформить или снять подписку, не нарушая идущий обход.

Время жизни сервисов организовано через иерархию областей жизни (LifetimeScope). Каждая область конфигурирует собственный контейнер зависимостей – реестр, который создаёт сервисы и передаёт их друг другу через параметры конструкторов. Подход реализован библиотекой VContainer – легковесной и широко распространённой в Unity-проектах; она снимает ручную сборку сервисов и поддерживает вложенные области без дополнительного кода [ИСТ: VContainer]. Корневая область RootLifetimeScope создаётся при запуске, переживает любые смены сцен и содержит сервисы времени жизни приложения; к ним относятся уже знакомые PathProvider, AppStorage и EventBus, а также библиотеки ассетов и конвейеры импорта и экспорта. Сценная область создаётся вместе со сценой режима и уничтожается при выходе из него; контейнер сценной области вложен в корневой, поэтому сценный сервис может зависеть от корневого, но не наоборот. Правило направления зависимостей видно на знакомом примере: сценный SceneAutoSaver получает через конструктор корневые AppStorage и EventBus, тогда как обратная связь – корневой сервис, удерживающий сценный, – исключена самой иерархией. Одновременно существует ровно одна сценная область. Состав областей приведён в Таблице 3.1.

Таблица 3.1 – Области жизни приложения и их ключевые сервисы

| Область жизни | Время существования | Ключевые сервисы |

Регистрация сервисов выполняется в методе Configure соответствующей области; характер записи виден из фрагмента корневой области в Листинге 3.12, полный текст скрипта приведён в Приложении Б (Листинг Б.3).

Листинг 3.12 – Скрипт RootLifetimeScope.cs (Фрагмент регистрации сервисов)
```csharp
protected override void Configure(IContainerBuilder builder)
{
    builder.Register<PathProvider>(Lifetime.Singleton);
    builder.Register<AppStorage>(Lifetime.Singleton);
    builder.Register<EventBus>(Lifetime.Singleton);
    builder.Register<SceneContext>(Lifetime.Singleton);
    builder.RegisterInstance(_transitionGraph);

    builder.RegisterEntryPoint<ImportedAssetLibrary>(Lifetime.Singleton).AsSelf();
    builder.RegisterEntryPoint<SavedAssetLibrary>(Lifetime.Singleton).AsSelf();
    builder.Register<AssetRegistry>(Lifetime.Singleton).As<IAssetRegistry>();

    builder.Register<GltfAssetImporter>(Lifetime.Singleton).As<IAssetImporter>();
    builder.Register<ImageAssetImporter>(Lifetime.Singleton).As<IAssetImporter>();
    builder.RegisterEntryPoint<ImportPipeline>(Lifetime.Singleton).AsSelf();

    builder.RegisterEntryPoint<SceneExporter>(Lifetime.Singleton).AsSelf();
    // ...
}
```

В листинге соседствуют два способа регистрации, и различие между ними определяется ролью сервиса. Большинство сервисов пассивны: такой сервис создаётся контейнером в момент, когда понадобится другому объекту, и работает только в ответ на обращения – для этого достаточно обычного Register. Однако часть сервисов должна действовать самостоятельно. Видная в том же листинге библиотека импортированных ассетов ImportedAssetLibrary обязана прочитать свой файл с диска сразу при запуске приложения – её к этому никто не побуждает. Компонентам Unity подобные моменты жизни предоставляет движок, вызывая их методы Start и Update, но сервисы каркаса компонентами не являются, и движок о них не знает. Для таких сервисов предусмотрена регистрация RegisterEntryPoint: жизненный цикл берёт на себя контейнер и сам вызывает сервис в нужные моменты. О том, какие именно моменты ему нужны, сервис заявляет реализацией интерфейса: IStartable означает вызов при построении области, ITickable – вызов каждый кадр. По этой же схеме работает, например, SceneDirtyTracker: контейнер сценной области вызывает его метод Start, в котором сервис оформляет подписки на сообщения. Покадровые вызовы применяются при воспроизведении анимации.

Из-за того, что наборы сервисов в сценных областях различаются, сервисам корневой области нельзя удерживать прямые ссылки на сценные сервисы: после смены режима такая ссылка указывала бы на уничтоженный объект. Посредником служит корневой сервис SceneContext – фасад, через который постоянные потребители, прежде всего панели интерфейса, обращаются к сервисам текущей сцены. При старте сценной области класс SceneContextBinder заполняет фасад ссылками на граф сцены, выделение и анимационные сервисы и публикует сообщение SceneContextChangedEvent; при уничтожении области фасад очищается с повторной публикацией сообщения. Заполнение выполняется защищённо: сервис, не зарегистрированный в текущей области, оставляет в фасаде пустую ссылку. Так, в песочнице анимационные сервисы отсутствуют, поэтому потребитель проверяет доступность именно того сервиса, который собирается использовать, а не факт загруженной сцены.

Третья задача каркаса – переключение режимов. Каждому режиму соответствует собственная сцена Unity и сценная область; с режимом связаны и доступные пользователю возможности: состав панелей интерфейса и кнопок навигации определяется режимом и перестраивается при его смене – подробно это поведение разбирается при описании системы интерфейсов. Сменой режимов управляет класс ModeOrchestrator: он проверяет допустимость перехода по графу ModeTransitionGraph и делегирует фактическую перезагрузку сцены исполнителю переходов. Граф задан как ассет с перечнем разрешённых пар: меню соединено двусторонними переходами с редактированием и песочницей, прямой переход между редактированием и песочницей не предусмотрен (Рисунок 3.5). Полный текст оркестратора приведён в Листинге 3.13.

Листинг 3.13 – Скрипт ModeOrchestrator.cs
```csharp
public class ModeOrchestrator
{
    private readonly EventBus            _bus;
    private readonly ModeTransitionGraph _graph;
    private readonly ISceneTransition    _transition;

    private AppMode _current = AppMode.MainMenu;
    public AppMode CurrentMode => _current;

    public ModeOrchestrator(EventBus bus, ModeTransitionGraph graph,
                            ISceneTransition transition)
    {
        _bus        = bus;
        _graph      = graph;
        _transition = transition;
    }

    public void TransitionTo(AppMode target)
    {
        if (_current == target) return;
        if (_transition.IsTransitioning) return;
        if (!_graph.IsAllowed(_current, target))
        {
            Debug.LogWarning($"Transition {_current} → {target} not allowed");
            return;
        }

        var prev = _current;
        _current = target;

        _bus.Publish(new ModeExitingEvent { From = prev, To = target });

        _transition.Load(SceneNameFor(target), () =>
            _bus.Publish(new ModeChangedEvent
                { PreviousMode = prev, CurrentMode = target }));
    }

    private static string SceneNameFor(AppMode mode) => mode switch
    {
        AppMode.MainMenu  => "MainMenu",
        AppMode.VrEditing => "VrEditing",
        AppMode.Sandbox   => "Sandbox",
        _                 => null,
    };
}
```

Порядок событий внутри перехода устроен под нужды сохранения данных. Сообщение ModeExitingEvent публикуется до начала загрузки новой сцены, пока прежняя сцена и её область жизни ещё существуют, – именно на это сообщение опирается автосохранение, описанное в пункте 3.1.4. Сообщение ModeChangedEvent публикуется после того, как новая сцена загружена и её область построена, поэтому получатели могут сразу обращаться к новому окружению.

Фактический переход выполняет класс SceneTransitionRunner, приведённый в Листинге 3.14.

Листинг 3.14 – Скрипт SceneTransitionRunner.cs
```csharp
public class SceneTransitionRunner : MonoBehaviour, ISceneTransition
{
    [SerializeField] private HeadFade _fade;

    public bool IsTransitioning { get; private set; }

    public void Load(string sceneName, Action onLoaded)
    {
        if (IsTransitioning || string.IsNullOrEmpty(sceneName)) return;
        IsTransitioning = true;
        StartCoroutine(RunRoutine(sceneName, onLoaded));
    }

    public void LoadInitial(string sceneName, Action onLoaded)
    {
        if (_fade != null) _fade.SetAlphaImmediate(1f);
        Load(sceneName, onLoaded);
    }

    private IEnumerator RunRoutine(string sceneName, Action onLoaded)
    {
        if (_fade != null) yield return StartCoroutine(_fade.FadeRoutine(1f));

        var op = SceneManager.LoadSceneAsync(sceneName, LoadSceneMode.Single);
        if (op != null) yield return op;

        onLoaded?.Invoke();

        if (_fade != null) yield return StartCoroutine(_fade.FadeRoutine(0f));

        IsTransitioning = false;
    }
}
```

Переход построен как корутина из четырёх шагов: обзор плавно затемняется, сцена загружается в одиночном режиме, замещая прежнюю, вызывается обратный вызов с публикацией ModeChangedEvent, затемнение снимается. Затемнение решает две задачи. Асинхронная загрузка сцены не гарантирует ровной частоты кадров и может сопровождаться короткими подвисаниями; за чёрным экраном эти артефакты не видны. Одновременно затемнение служит пользователю наглядным маркером смены режима: окружение не подменяется на глазах, а проявляется уже новым. Флаг IsTransitioning защищает корутину от повторного входа – запрос перехода во время идущего перехода отбрасывается.

Тем же механизмом приложение пользуется при запуске. Сцена начальной загрузки содержит постоянный корень PersistentRoot, под которым живут XR-риг пользователя с закреплёнными на нём панелями, исполнитель переходов и корневая область жизни. Точка входа AppBootstrap помечает этот корень как переживающий смены сцен и запускает первую загрузку главного меню методом LoadInitial. После первой одиночной загрузки сцена начальной загрузки выгружается – остаётся только постоянный корень. Полный текст точки входа приведён в Листинге 3.15.

Листинг 3.15 – Скрипт AppBootstrap.cs
```csharp
public class AppBootstrap : MonoBehaviour
{
    private const string MAIN_MENU_SCENE = "MainMenu";

    [SerializeField] private GameObject            _persistentRoot;
    [SerializeField] private SceneTransitionRunner _transitionRunner;

    private void Start()
    {
        Cursor.visible   = false;
        Cursor.lockState = CursorLockMode.Locked;

        if (_persistentRoot != null) DontDestroyOnLoad(_persistentRoot);

        if (_transitionRunner != null)
            _transitionRunner.LoadInitial(MAIN_MENU_SCENE, null);
        else
            Debug.LogError("AppBootstrap: _transitionRunner not assigned — "
                + "first scene will not load.");
    }
}
```

Собранная система режимов выглядит так. Главное меню служит отправной точкой: из него пользователь открывает сцены и настройки. Режим редактирования – основное рабочее пространство: в его области жизни, как видно из Таблицы 3.1, зарегистрированы анимационные сервисы и автосохранение. Песочница повторяет редактирование за вычетом анимации и сохранения: размещение объектов, выделение и работа с костями доступны в полном объёме, но сцена временна и, как отмечалось в пункте 3.1.1, на диск не записывается – это пространство свободного эксперимента, не угрожающего сохранённым сценам.

В совокупности каркас задаёт правила, по которым существует остальное приложение: шина сообщений связывает модули, не знакомя их друг с другом; иерархия областей жизни выдаёт каждому сервису ровно то время существования, которое отвечает его роли; оркестратор переводит приложение между режимами, не теряя пользовательских данных.
