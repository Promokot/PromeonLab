# 3.2.1 Архитектурный каркас приложения

> Статус: черновик на ревью.
> Нумерация сквозная в пределах раздела 3, реестр: numbering-registry.md.

Каркас приложения решает три задачи: связывает модули между собой, управляет временем жизни сервисов и переключает режимы работы. Для связи модулей используется обмен сообщениями через общую шину, для управления временем жизни – внедрение зависимостей (Dependency Injection, DI) на основе библиотеки VContainer [ИСТ: VContainer], для переключения режимов – оркестратор с графом разрешённых переходов. Эти три механизма рассматриваются по порядку.

Шину сообщений реализует класс EventBus. Сообщением служит типизированная структура с суффиксом Event в имени – такие сообщения уже встречались при описании автосохранения: SceneModifiedEvent, SceneOpenedEvent. Модуль-отправитель публикует структуру методом Publish, модули-получатели заранее оформляют подписку методом Subscribe; отправитель не знает своих получателей, и наоборот [ИСТ: паттерн Publish-Subscribe]. Прямые вызовы между модулями тем самым исключаются: добавление нового получателя сообщения не требует правок в отправителе. Полный текст класса приведён в Листинге 3.11.

Листинг 3.11 – Скрипт EventBus.cs
```csharp
public class EventBus
{
    private readonly Dictionary<Type, List<object>> _handlers = new();

    public void Subscribe<T>(Action<T> handler) where T : struct
    {
        var type = typeof(T);
        if (!_handlers.ContainsKey(type))
            _handlers[type] = new List<object>();
        _handlers[type].Add(handler);
    }

    public void Unsubscribe<T>(Action<T> handler) where T : struct
    {
        if (_handlers.TryGetValue(typeof(T), out var list))
            list.Remove(handler);
    }

    public void Publish<T>(T message) where T : struct
    {
        if (!_handlers.TryGetValue(typeof(T), out var list)) return;
        foreach (var handler in list.ToArray())
            ((Action<T>)handler).Invoke(message);
    }
}
```

Реализация компактна: словарь сопоставляет тип сообщения списку обработчиков. Существенная деталь скрыта в методе Publish: перед обходом список обработчиков копируется в массив, поэтому обработчик может в ответ на сообщение оформить или снять подписку, не нарушая идущий обход.

Время жизни сервисов организовано через иерархию областей жизни (LifetimeScope) [ИСТ: VContainer]. Каждая область конфигурирует собственный контейнер зависимостей – реестр, который создаёт сервисы и передаёт их друг другу через параметры конструкторов. Корневая область RootLifetimeScope создаётся при запуске, переживает любые смены сцен и содержит сервисы времени жизни приложения; к ним относятся уже знакомые PathProvider, AppStorage и EventBus, а также библиотеки ассетов и конвейеры импорта и экспорта. Сценная область создаётся вместе со сценой режима и уничтожается при выходе из него; контейнер сценной области вложен в корневой, поэтому сценный сервис может зависеть от корневого, но не наоборот. Одновременно существует ровно одна сценная область. Состав областей приведён в Таблице 3.1.

Таблица 3.1 – Области жизни приложения и их ключевые сервисы

| Область жизни | Время существования | Ключевые сервисы |

| RootLifetimeScope | Всё время работы приложения | EventBus, PathProvider, AppStorage, SceneContext, ModeOrchestrator, библиотеки ассетов, ImportPipeline, SceneExporter, VrKeyboard, UserPanel |
| MainMenuSceneScope | Режим главного меню | SceneGraph, панели меню и выбора сцен |

| SandboxSceneScope | Режим песочницы | SceneGraph, SelectionManager, AssetSpawner, BoneEditMode |

Регистрация сервисов выполняется в методе Configure соответствующей области; характер записи виден из фрагмента корневой области в Листинге 3.12, полный текст скрипта приведён в Приложении Б (Листинг Б.3).

Из-за того, что наборы сервисов в сценных областях различаются, сервисам корневой области нельзя удерживать прямые ссылки на сценные сервисы: после смены режима такая ссылка указывала бы на уничтоженный объект. Посредником служит корневой сервис SceneContext – фасад, через который постоянные потребители, прежде всего панели интерфейса, обращаются к сервисам текущей сцены. При старте сценной области класс SceneContextBinder заполняет фасад ссылками на граф сцены, выделение и анимационные сервисы и публикует сообщение SceneContextChangedEvent; при уничтожении области фасад очищается с повторной публикацией сообщения. Заполнение выполняется защищённо: сервис, не зарегистрированный в текущей области, оставляет в фасаде пустую ссылку. Так, в песочнице анимационные сервисы отсутствуют, поэтому потребитель проверяет доступность именно того сервиса, который собирается использовать, а не факт загруженной сцены.

Третья задача каркаса – переключение режимов. Приложение работает в одном из трёх режимов: главное меню, редактирование и песочница; каждому режиму соответствует собственная сцена Unity и сценная область. Сменой режимов управляет класс ModeOrchestrator: он проверяет допустимость перехода по графу ModeTransitionGraph и делегирует фактическую перезагрузку сцены исполнителю переходов. Граф задан как ассет с перечнем разрешённых пар: меню соединено двусторонними переходами с редактированием и песочницей, прямой переход между редактированием и песочницей не предусмотрен (Рисунок 3.5). Полный текст класса приведён в Листинге 3.13.

Листинг 3.13 – Скрипт ModeOrchestrator.cs
```csharp
public class ModeOrchestrator
{
    private readonly EventBus            _bus;
    private readonly ModeTransitionGraph _graph;
    private readonly ISceneTransition    _transition;

    private AppMode _current = AppMode.MainMenu;
    public AppMode CurrentMode => _current;

    public ModeOrchestrator(EventBus bus, ModeTransitionGraph graph,
                            ISceneTransition transition)
    {
        _bus        = bus;
        _graph      = graph;
        _transition = transition;
    }

    public void TransitionTo(AppMode target)
    {
        if (_current == target) return;
        if (_transition.IsTransitioning) return;
        if (!_graph.IsAllowed(_current, target))
        {
            Debug.LogWarning($"Transition {_current} → {target} not allowed");
            return;
        }

        var prev = _current;
        _current = target;

        _bus.Publish(new ModeExitingEvent { From = prev, To = target });

        _transition.Load(SceneNameFor(target), () =>
            _bus.Publish(new ModeChangedEvent
                { PreviousMode = prev, CurrentMode = target }));
    }

    private static string SceneNameFor(AppMode mode) => mode switch
    {
        AppMode.MainMenu  => "MainMenu",
        AppMode.VrEditing => "VrEditing",
        AppMode.Sandbox   => "Sandbox",
        _                 => null,
    };
}
```

Порядок событий внутри перехода устроен под нужды сохранения данных. Сообщение ModeExitingEvent публикуется до начала загрузки новой сцены, пока прежняя сцена и её область жизни ещё существуют, – именно на это сообщение опирается автосохранение, описанное в пункте 3.1.4. Сообщение ModeChangedEvent публикуется после того, как новая сцена загружена и её область построена, поэтому получатели могут сразу обращаться к новому окружению. Сам переход выполняет класс SceneTransitionRunner: экран плавно затемняется, сцена загружается в одиночном режиме, замещая прежнюю, затем затемнение снимается. Затемнение скрывает от пользователя момент подмены окружения – резкая смена картинки в гарнитуре воспринималась бы как рывок. Повторный запрос перехода во время идущего перехода отбрасывается защитой от повторного входа.

Режим песочницы занимает в этой схеме особое место. Он использует тот же набор механизмов – собственную сцену, область жизни и граф сцены, – но работает с временной сценой, которая, как отмечалось в пункте 3.1.1, не записывается на диск. Песочница служит пространством свободного эксперимента: пользователь размещает объекты и пробует инструменты, не опасаясь за сохранённые сцены.
