> [!info] Назначение
> `SceneAnimationData`, `ActionContainer`, `AnimTrackData`, `AnimKeyData` — иерархия сериализуемых структур, описывающих анимационные данные сцены. Относятся к подсистеме **Animation**. Поля без методов приведены в **Листинге 3.5** раздела 3.1.2 ВКР.

### Обзор

##### Роль и место

`SceneAnimationData` — корень, записываемый [[AnimationStorage]] в `animation.json`. Хранит общую для сцены частоту кадров `Fps` и список контейнеров действий `Containers`. Каждый `ActionContainer` привязан к одной ноде через `OwnerNodeId` и содержит список дорожек `List<AnimTrackData>`. Дорожка `AnimTrackData` идентифицирует конкретный узел трансформации (саму ноду или отдельную кость) и хранит упорядоченный список ключей `AnimKeyData`. Ключ фиксирует номер кадра и полный набор трансформаций (позиция + поворот + масштаб).

Все четыре типа помечены `[Serializable]`; поля `public` по той же причине, что и у структур сцены — требование `JsonUtility`. Логика CRUD-операций над этой иерархией сосредоточена в `AnimationAuthoring`; воспроизведение — в `AnimationPlaybackSampler`.

##### Ключевые методы

`SceneAnimationData`:
- `FindByOwner(ownerNodeId)` — линейный поиск контейнера по владельцу.
- `CreateContainer(ownerNodeId, ...)` — идемпотентно: возвращает существующий или создаёт новый.
- `RemoveContainer(ownerNodeId)` — удаляет контейнер с конца списка (обход от конца во избежание сдвига индексов).

`ActionContainer`:
- `GetOrCreateTrack(nodeId)` — идемпотентно создаёт дорожку.
- `HasAnyKeyAtFrame(frame)` — проверяет наличие ключа на кадре хотя бы в одной дорожке.
- `TruncateToTotalFrames()` — обрезает ключи, выходящие за `TotalFrames`.

`AnimTrackData`:
- `UpsertKey(frame, pos, rot, scale)` — заменяет существующий ключ или добавляет новый, сортирует список.
- `RemoveKey(frame)` / `TrimKeysAfter(maxFrame)` / `HasKey(frame)`.

### Разбор кода

##### OwnerNodeId vs. NodeId дорожки

```csharp
public class ActionContainer
{
    public string OwnerNodeId;
    ...
    public List<AnimTrackData> Tracks = new();
}

public class AnimTrackData
{
    public string NodeId;
    ...
}
```

> Два разных поля для двух разных уровней. `OwnerNodeId` в контейнере — это нода-«хозяин» действия (объект сцены в целом). `NodeId` в дорожке — конкретный узел трансформации: либо тот же `OwnerNodeId` (для трансформации самого объекта), либо идентификатор кости прокси-рига (для покостной анимации). Такое разделение позволяет одному контейнеру содержать дорожки как для объекта, так и для всех его костей одновременно.

##### Fps на уровне контейнера — резервное поле

```csharp
public class ActionContainer
{
    public int Fps = 24;
    ...
}

public class SceneAnimationData
{
    public int Fps = 24;
    ...
}
```

> Оба поля называются `Fps` и имеют одинаковое значение по умолчанию. По нарративу ВКР, `Fps` контейнера — «резерв на случай загрузки данных без общесценовой частоты». Это страховка обратной совместимости: если в будущем схема поменяется или файл повреждён так, что `SceneAnimationData.Fps` недоступен, можно использовать значение из контейнера. В текущей реализации `AnimationStorage.LoadAsync` при `loaded.Fps <= 0` читает именно `loaded.Containers[0].Fps` как запасной вариант.

##### UpsertKey — сортировка после вставки

```csharp
public void UpsertKey(int frame, Vector3 pos, Quaternion rot, Vector3 scale)
{
    for (int i = 0; i < Keys.Count; i++)
    {
        if (Keys[i].Frame != frame) continue;
        Keys[i] = new AnimKeyData { Frame = frame, Position = pos, Rotation = rot, Scale = scale };
        return;
    }
    Keys.Add(new AnimKeyData { Frame = frame, Position = pos, Rotation = rot, Scale = scale });
    Keys.Sort((a, b) => a.Frame.CompareTo(b.Frame));
}
```

> Сортировка вызывается только при добавлении нового ключа (путь `Keys.Add + Keys.Sort`), но не при обновлении существующего (путь `Keys[i] = ... return`). Это верно: замена ключа с тем же номером кадра не меняет порядка. `Keys.Sort` после каждой вставки — O(n log n), но список ключей в реальных анимациях небольшой, поэтому это приемлемо. Хранение в отсортированном порядке критично для `AnimationClipBaker`, который интерпретирует ключи как временную последовательность.

##### TruncateToTotalFrames — обход от конца

```csharp
public void TruncateToTotalFrames()
{
    for (int i = Tracks.Count - 1; i >= 0; i--)
        Tracks[i].TrimKeysAfter(TotalFrames);
}
```

> Обход дорожек не требует обратного порядка (дорожки не удаляются), но `TrimKeysAfter` внутри обходит `Keys` от конца, чтобы `RemoveAt(i)` не сдвигал индексы не-удалённых элементов вперёд. Это стандартный паттерн удаления из `List<T>` без создания временной коллекции.

##### AnimKeyData — все трансформации в одном ключе

```csharp
[Serializable]
public class AnimKeyData
{
    public int        Frame;
    public Vector3    Position;
    public Quaternion Rotation;
    public Vector3    Scale;
}
```

> Ключ хранит сразу pos + rot + scale, то есть дорожка не разделяется на отдельные кривые по свойствам (как это делает `AnimationCurve` в Unity). Это упрощает операции: постановка/удаление ключа — атомарная операция над одним элементом списка. Недостаток — нельзя поставить ключ только по позиции без фиксации текущего поворота, но для данного приложения это приемлемо.

### К защите

> [!question] Вероятные вопросы
>
> **В:** Чем отличается `OwnerNodeId` контейнера от `NodeId` дорожки?
> О: `OwnerNodeId` — идентификатор ноды-владельца в графе сцены (объект целиком). `NodeId` дорожки — конкретный узел трансформации: это может быть та же нода (для трансформации объекта) или идентификатор кости прокси-рига (для покостной анимации). Один контейнер содержит дорожки как для объекта, так и для всех его костей одновременно.
>
> **В:** Зачем у `ActionContainer` есть собственный `Fps`, если у `SceneAnimationData` тоже есть `Fps`?
> О: `Fps` контейнера — резервное поле для обратной совместимости. При загрузке файла, если общесценовой `Fps` недоступен или равен нулю, `AnimationStorage` использует `Containers[0].Fps` как запасное значение. В нормальной работе используется только `SceneAnimationData.Fps`.
>
> **В:** Почему ключ хранит сразу позицию, поворот и масштаб, а не отдельные кривые?
> О: Это упрощает модель данных: постановка, перенос и удаление ключа — атомарная операция над одним элементом списка. Нельзя случайно получить ключ только по позиции с пропущенным поворотом. Платой является то, что нельзя поставить ключ только по одному свойству — для данного приложения это ограничение приемлемо.
>
> **В:** Почему ключи в `AnimTrackData.Keys` отсортированы и кто это гарантирует?
> О: Сортировку обеспечивает `UpsertKey`: после каждого добавления нового ключа вызывается `Keys.Sort`. При обновлении существующего ключа сортировка не нужна — порядок не меняется. `AnimationClipBaker` полагается на отсортированный порядок при построении `AnimationCurve` для воспроизведения.
