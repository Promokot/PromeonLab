> [!info] Назначение
> `SceneData`, `NodeData`, `BonePose` — сериализуемые структуры, описывающие граф сцены с иерархией объектов и позами костей. Относятся к подсистеме **StorageCore**. Поля приведены в **Листинге 3.4** раздела 3.1.2 ВКР.

### Обзор

##### Роль и место

`SceneData` — корневой объект, который [[SceneSerializer]] записывает в `scene.json`. Содержит плоский список `List<NodeData>` вместо вложенного дерева: иерархия объектов задаётся полем `ParentNodeId`, а не вложенностью структур. `NodeData` представляет один объект сцены: его трансформацию, ссылку `AssetRef` на запись в библиотеке и список поз костей `BonePoses` (пустой для не-скелетных нод). `BonePose` хранит локальные трансформации одной кости скелета.

Все три типа помечены `[Serializable]` и используются `JsonUtility`. Поля — `public`, что является архитектурным исключением: у классов с поведением поля должны быть `[SerializeField] private`, но чистые data-классы, сериализуемые `JsonUtility`, вынуждены иметь `public` поля, поскольку `JsonUtility` не читает `private`-поля без `[SerializeField]`.

##### Ключевые методы

Все три класса — чистые данные без методов. Логика работы со списком `Nodes` инкапсулирована в `SceneGraph` (добавление, поиск, снимок). Сериализация — [[SceneSerializer]]. Загрузка/сохранение — [[AppStorage]].

### Разбор кода

##### Плоский список вместо вложенного дерева

```csharp
public class SceneData
{
    ...
    public List<NodeData> Nodes = new();
}

public class NodeData
{
    ...
    public string ParentNodeId;
    ...
}
```

> Иерархия выражена ссылкой `ParentNodeId` (пустая строка = корень). Альтернатива — вложенные `List<NodeData> Children` — выглядела бы «естественнее», но усложнила бы сериализацию (рекурсивные структуры), обход (рекурсивные итераторы) и операции перемещения между родителями (поиск в дереве вместо словаря по ID). Плоский список позволяет строить `Dictionary<string, SceneNode>` за один проход и восстанавливать иерархию за второй проход — это и делает `SceneGraph` при загрузке.

##### BonePoses — инлайн в NodeData, не отдельный файл

```csharp
public class NodeData
{
    ...
    public List<BonePose> BonePoses = new(); // empty for non-rig nodes and pre-v3 scenes
}
```

> До v3 схемы позы костей лежали в отдельных файлах (`scenes/{id}/Rigs/`, `scenes/{id}/Poses/`). В v3 они перенесены инлайн в `NodeData`. Преимущество: граф сцены и позы костей читаются за одну операцию, нет риска рассинхрона файлов. Недостаток: `scene.json` растёт пропорционально числу костей. Комментарий `// empty for non-rig nodes and pre-v3 scenes` — документирует граничный случай, который обрабатывается миграцией в `SceneSerializer`.

##### public-поля у JsonUtility-данных

```csharp
[Serializable]
public class SceneData
{
    public int            SchemaVersion = 3;
    public string         SceneId;
    ...
}
```

> `JsonUtility.ToJson`/`FromJson` читают только `public`-поля или `private`-поля с `[SerializeField]`. Для чистых data-классов принято использовать `public`-поля — это исключение из общего правила «`[SerializeField] private` для `MonoBehaviour`». Классы `SceneData`, `NodeData`, `BonePose` не несут поведения и никогда не становятся компонентами объектов сцены, поэтому public-поля здесь корректны.

##### schemaVersion = 3 как дефолт

```csharp
public int SchemaVersion = 3;
```

> Значение 3 — это дефолт нового объекта, создаваемого в коде (`new SceneData { ... }`). При десериализации `JsonUtility` перезапишет это поле значением из файла. Если в файле поле отсутствует (очень старый формат), `JsonUtility` оставит дефолт — но это неверно: старый файл должен читаться как v1. Именно поэтому [[SceneSerializer]] проверяет `data.SchemaVersion < 2` и `< 3` после десериализации, а не полагается на дефолт.

### К защите

> [!question] Вероятные вопросы
>
> **В:** Почему иерархия объектов задана `ParentNodeId`, а не вложенными списками?
> О: Плоский список с ссылочной иерархией проще сериализовать (нет рекурсии), проще обходить (линейный проход → словарь по ID) и проще модифицировать (перемещение ноды = смена одного строкового поля). `SceneGraph` восстанавливает иерархию за два прохода: сначала строит словарь всех нод, затем связывает родитель-потомок по `ParentNodeId`.
>
> **В:** Почему у `SceneData` и `NodeData` `public` поля, хотя в правилах проекта требуется `[SerializeField] private`?
> О: Это архитектурное исключение для data-классов, сериализуемых `JsonUtility`: сериализатор не читает `private`-поля без `[SerializeField]`, а добавлять атрибут к каждому полю чистой структуры — лишний шум. У классов без поведения `public`-поля безопасны. CLAUDE.md явно перечисляет это исключение: «`[Serializable]` data classes serialized by JsonUtility may use public fields».
>
> **В:** Что изменилось в v3 схемы и зачем нужна миграция?
> О: В v3 добавлены `BonePoses` — список поз костей инлайн в каждой ноде. В файлах v1/v2 это поле отсутствует, поэтому `JsonUtility` при десериализации оставит его `null`. Миграция в `SceneSerializer` инициализирует `BonePoses = new List<BonePose>()` для каждой ноды, чтобы остальной код не проверял `null` везде.
>
> **В:** Почему `BonePoses` перенесли инлайн, а не оставили в отдельном файле?
> О: Инлайн-хранение исключает рассинхрон: граф сцены и позы костей читаются и записываются атомарно как один файл. При отдельном файле поз возможна ситуация, когда `scene.json` обновился, а файл поз — нет, что дало бы некорректное состояние после краша или прерванной записи.
