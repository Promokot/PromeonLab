> [!info] Назначение
> `EventBus` — типизированная шина сообщений, единственный легальный способ связи между подсистемами. Живёт в корневой области (`RootLifetimeScope`), существует всё время работы приложения. Листинг 3.11 приводит полный текст класса.

### Обзор

##### Роль и место

`EventBus` — реализация паттерна [[Паттерн Publish-Subscribe]] поверх словаря `Dictionary<Type, List<object>>`. Один экземпляр регистрируется синглтоном в [[RootLifetimeScope]] и впрыскивается конструктором во все подсистемы. Согласно соглашениям проекта, **прямые вызовы методов между подсистемами запрещены**: все уведомления идут через `Publish<T>`. Тип сообщения ограничен `where T : struct` — все сообщения представляют собой `struct`-типы с суффиксом `Event` (например, `SceneModifiedEvent`, `ModeChangedEvent`).

Область не одна: каждая `LifetimeScope` в принципе может иметь свой `EventBus`, но в данном проекте используется единственный корневой экземпляр, доступный сквозь всю иерархию областей.

##### Ключевые методы

| Метод | Что делает |
|---|---|
| `Subscribe<T>(Action<T>)` | Регистрирует обработчик для типа `T`; создаёт запись в словаре при первой подписке |
| `Unsubscribe<T>(Action<T>)` | Удаляет делегат из списка по значению (`List.Remove`) |
| `Publish<T>(T message)` | Рассылает сообщение всем подписчикам; безопасен к изменению списка во время обхода |

### Разбор кода

##### Словарь с type erasure

```csharp
private readonly Dictionary<Type, List<object>> _handlers = new();
```

> Тип ключа — `System.Type`, получаемый в рантайме через `typeof(T)`. Значение — `List<object>`: сохранить делегаты разных типов (`Action<SceneOpenedEvent>`, `Action<ModeChangedEvent>`) в одном поле без бокса делегата нельзя иначе, чем стереть конкретный тип до `object`. Этот приём называют **type erasure** — конкретная информация о `T` теряется в коллекции, но восстанавливается при извлечении через каст. Именно поэтому словарь `readonly`: ссылка на него никогда не меняется, мутируются только внутренние `List<object>`.

##### Subscribe — lazy initialization

```csharp
public void Subscribe<T>(Action<T> handler) where T : struct
{
    var type = typeof(T);
    if (!_handlers.ContainsKey(type))
        _handlers[type] = new List<object>();
    _handlers[type].Add(handler);
}
```

> `where T : struct` — ограничение типа на уровне компилятора, а не рантайма. Оно даёт три гарантии: (1) `T` никогда не `null` (значимый тип); (2) сравнение `typeof(T)` всегда точное — нет наследования, нет полиморфизма, нет двусмысленности в словаре; (3) разные типы `struct` не спутать, даже если они одинакового размера. Инициализация списка ленивая — запись в словаре появляется только при первой подписке на данный тип. `ContainsKey` + `[]` — два обращения к словарю вместо одного `TryGetValue`; для пути «уже есть» это чуть дороже, но принципиально не важно: `Subscribe` вызывается редко (при старте области, не в горячем пути).

##### Unsubscribe — List.Remove по значению

```csharp
public void Unsubscribe<T>(Action<T> handler) where T : struct
{
    if (_handlers.TryGetValue(typeof(T), out var list))
        list.Remove(handler);
}
```

> `list.Remove(handler)` сравнивает делегаты **по значению** (`Delegate.Equals`). Для лямбд, захваченных в замыкание, каждая `new Action<T>(...)` — отдельный объект, и `Remove` не найдёт «похожую» лямбду — только ту же самую ссылку. Правило проекта: подписчик сохраняет делегат в поле и передаёт тот же экземпляр в `Unsubscribe`. Если тип ещё не зарегистрирован, `TryGetValue` вернёт `false` — тихо ничего не делаем (не исключение, не варнинг).

##### Publish — снимок коллекции (ToArray)

```csharp
public void Publish<T>(T message) where T : struct
{
    if (!_handlers.TryGetValue(typeof(T), out var list)) return;
    foreach (var handler in list.ToArray())
        ((Action<T>)handler).Invoke(message);
}
```

> **`list.ToArray()`** — не косметика, а защита от `InvalidOperationException`. `_handlers[type]` — живой `List<object>`. Если обработчик внутри `Invoke` вызовет `Subscribe` или `Unsubscribe` на тот же тип (например, панель отписывается, получив событие закрытия), коллекция изменится прямо во время `foreach` → исключение «коллекция изменена». `ToArray()` снимает **мгновенную копию** списка перед обходом: новая подписка появится в исходном `List`, но итератор её не увидит. Цена — выделение временного массива на каждый `Publish`. При частых событиях (например, `FrameChangedEvent` на 90 fps) это создаёт мусор для GC. В текущем проекте принято как допустимый компромисс ради простоты.
>
> **Каст `(Action<T>)handler`** обязателен: `handler` имеет тип `object` (type-erased). Возможен ClassCastException только при ошибке в `Subscribe` (неверный тип в список) — в правильно написанном коде такого нет, потому что в список типа `T` попадают только `Action<T>` (дженерик-ограничение не позволяет добавить что-то другое).
>
> Ранний `return` при отсутствии ключа предотвращает создание пустого массива и обход — микрооптимизация, но важная для частых событий без подписчиков.

### К защите

> [!question] Вероятные вопросы
>
> **В:** Зачем `ToArray()` — разве `foreach` по `List` не безопасен?
> **О:** `foreach` по `List<T>` в C# использует структурный `Enumerator`, который проверяет версию коллекции при каждом шаге и выбрасывает `InvalidOperationException`, если список изменился. Если обработчик во время `Invoke` вызовет `Subscribe`/`Unsubscribe` на тот же тип — список изменится прямо во время итерации. `ToArray()` снимает копию на момент начала рассылки, после чего обход идёт по неизменяемому массиву.
>
> **В:** Почему `List<object>`, а не строго типизированный список?
> **О:** Один словарь должен хранить обработчики для всех типов сообщений. `Action<SceneModifiedEvent>` и `Action<ModeChangedEvent>` — несовместимые типы, общий базовый тип у них только `object` (или `Delegate`). Стирание типа до `object` — единственный способ уложить их в один контейнер. При извлечении тип восстанавливается через каст `(Action<T>)`.
>
> **В:** Почему ограничение `where T : struct`?
> **О:** Три причины: (1) структура не может быть `null` — сообщение всегда существует; (2) структуры не наследуются, поэтому `typeof(T)` однозначно идентифицирует тип без полиморфных сюрпризов; (3) соглашение проекта: сообщения — лёгкие данные без поведения, структура лучше отражает этот семантику.
>
> **В:** Может ли один класс подписаться дважды на одно событие?
> **О:** Да, ничто не мешает вызвать `Subscribe` дважды с одним делегатом — он появится в списке дважды и получит событие дважды. Защиты от дублирования нет — ответственность на подписчике (обычно `Subscribe` вызывается один раз в `Start`/конструкторе и `Unsubscribe` — в `Dispose`).
>
> **В:** Есть ли порядок доставки?
> **О:** Порядок доставки совпадает с порядком подписки (FIFO внутри `List`). Порядок инициализации `IStartable` в VContainer детерминирован, но не специфицирован явно — полагаться на конкретный порядок доставки не следует.
>
> **В:** Что произойдёт, если ни одного подписчика нет?
> **О:** `TryGetValue` вернёт `false`, ранний `return` — событие молча теряется. Это нормальное поведение для системы без подписчиков (например, при закрытии области).
