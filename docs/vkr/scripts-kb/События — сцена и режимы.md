# События — сцена и режимы

> [!info] Назначение
> События жизненного цикла сцены и переключения режимов. Все — `public struct` без собственных конструкторов (см. [[Событийные структуры (Event)]]), рассылаются через [[EventBus]]. Папки `SceneComposition/Events/`, `Bootstrap/Events/`, `ModeOrchestrator/Events/`.

### Жизненный цикл сцены

- ##### SceneOpenedEvent
    - Поля: `string SceneId`. Конструирование: `new SceneOpenedEvent { SceneId = data.SceneId }`.
    - Публикует [[MainMenuPanel]] после загрузки сцены; слушают [[SceneGraph]], браузер ассетов — заполняют граф и сетку по новой сцене.
- ##### SceneClosedEvent
    - Маркер без полей: `public struct SceneClosedEvent { }` → `new SceneClosedEvent()`. Публикуется [[SceneAutoSaver]] только **после** успешной записи сцены на диск.
- ##### SceneModifiedEvent
    - Маркер без полей. Конструирование: `_bus.Publish(new SceneModifiedEvent())`.
    - Публикуется при любой мутации графа ([[SceneGraph]], [[InspectorPanel]] — добавление/удаление/перенос/переименование/правка трансформаций). Слушает [[SceneDirtyTracker]] — поднимает флаг несохранённого. См. [[Дебаунс записи]] (для анимации механизм другой).
- ##### SceneContextChangedEvent
    - Поля: `bool HasScene`. Конструирование: `new SceneContextChangedEvent { HasScene = _ctx.HasScene }`.
    - Сигнал **жизненного цикла DI**, а не данных: публикуется [[SceneContextBinder]] при заполнении фасада (`true`) и очистке (`false`). Слушают [[OutlinerPanel]], [[InspectorPanel]], [[AnimatorPanel]] — перестраивают/гасят UI, когда сценовые сервисы появляются/исчезают. Отличать от `SceneOpenedEvent` (данные сцены) и `ModeChangedEvent` (видимость панелей).

### Переключение режимов

- ##### ModeExitingEvent
    - Поля: `AppMode From`, `AppMode To`. Конструирование: `new ModeExitingEvent { From = prev, To = target }`.
    - Публикуется [[ModeOrchestrator]] **синхронно ДО** одиночной загрузки сцены — пока выходящая сцена и её область ещё живы. Это хук для [[SceneAutoSaver]]: захватить снимок графа, пока объекты не выгружены.
- ##### ModeChangedEvent
    - Поля: `AppMode PreviousMode`, `AppMode CurrentMode`. Конструирование: `new ModeChangedEvent { PreviousMode = prev, CurrentMode = target }`.
    - Публикуется [[SceneTransitionRunner]]/[[ModeOrchestrator]] **ПОСЛЕ** загрузки новой сцены. Слушает регионная модель UI ([[PanelRegionRouter]]) — перестраивает видимость панелей и кнопок. `AppMode`: `MainMenu, VrEditing, Sandbox, Debug`.
- ##### NodeRenamedEvent
    - Поля: `string NodeId`, `string NewName`. Конструирование: `new NodeRenamedEvent { NodeId = _bound.NodeId, NewName = finalName }`.
    - Публикует [[InspectorPanel]] при переименовании; слушает [[OutlinerPanel]] — точечно обновляет подпись строки без полной перестройки списка.

### К защите

> [!question] Вероятные вопросы
> **В: Почему два события на смену режима — Exiting и Changed?** О: Порядок. `ModeExitingEvent` летит синхронно, пока старая сцена жива (нужно успеть сохранить снимок); `ModeChangedEvent` — после загрузки новой сцены (когда уже можно перестраивать UI). Старая область к этому моменту уничтожена.
>
> **В: Зачем отдельный SceneContextChangedEvent, если есть SceneOpenedEvent?** О: Это сигнал DI-жизненного цикла (сценовые сервисы привязаны/отвязаны), а не данных сцены. Панели подписываются именно на него, чтобы знать, когда [[SceneContext]] можно безопасно читать.
>
> **В: Как заполняются поля этих событий?** О: Инициализатором объекта на месте публикации (`new X { … }`); собственных конструкторов нет (см. [[Событийные структуры (Event)]]).
>
> **В: SceneModifiedEvent без полей — зачем?** О: Получателю ([[SceneDirtyTracker]]) важен только факт изменения, не его детали — он лишь поднимает флаг «есть несохранённое».
