> [!info] Назначение
> `SceneDirtyTracker` — отслеживает флаг несохранённых изменений сцены, подписываясь на `SceneModifiedEvent` и `SceneOpenedEvent`. Относится к подсистеме **StorageCore**. Полный текст приведён в **Листинге 3.8** раздела 3.1.4 ВКР.

### Обзор

##### Роль и место

Регистрируется в сцен-скоупе `VrEditing`, реализует `IStartable, IDisposable` (VContainer-контракты для управления жизненным циклом). Подписывается на `EventBus` в `Start()` и отписывается в `Dispose()`. Хранит единственный бит состояния: `bool _isDirty`. Публичные потребители — навигационная логика (`CanNavigate()`) и ручное сбрасывание флага (`ClearDirty()`).

Класс намеренно минимален: не знает, что именно изменилось и как сохранять. Сохранением занимается [[SceneAutoSaver]], сигнализацией об изменениях — операции в `SceneGraph`, `AnimationAuthoring` и других подсистемах через `EventBus.Publish<SceneModifiedEvent>`.

##### Ключевые методы

| Метод | Суть |
|---|---|
| `Start()` | Подписывается на `SceneModifiedEvent` и `SceneOpenedEvent` |
| `Dispose()` | Отписывается от обоих событий |
| `IsDirty` | Геттер флага — только чтение снаружи |
| `CanNavigate()` | `!_isDirty` — используется для блокировки перехода из режима |
| `ClearDirty()` | Ручной сброс после явного сохранения |
| `OnModified(SceneModifiedEvent)` | `_isDirty = true` |
| `OnOpened(SceneOpenedEvent)` | `_isDirty = false` |

### Разбор кода

##### IStartable вместо конструктора для подписки

```csharp
public SceneDirtyTracker(EventBus bus) => _bus = bus;

public void Start()
{
    _bus.Subscribe<SceneModifiedEvent>(OnModified);
    _bus.Subscribe<SceneOpenedEvent>(OnOpened);
}
```

> Подписка в `Start()`, а не в конструкторе — не случайность. VContainer вызывает `IStartable.Start()` после того, как весь граф зависимостей скоупа построен. Подписка в конструкторе означала бы, что `SceneDirtyTracker` может получать события ещё до того, как другие сервисы (например, `SceneGraph`) инициализированы — это рецепт порядко-зависимых багов. `IStartable` — VContainer-контракт именно для «пост-инициализации».

##### Симметричная отписка в Dispose

```csharp
public void Dispose()
{
    _bus.Unsubscribe<SceneModifiedEvent>(OnModified);
    _bus.Unsubscribe<SceneOpenedEvent>(OnOpened);
}
```

> Подписка и отписка симметричны. Если не вызвать `Unsubscribe`, `EventBus` сохранит ссылку на `OnModified`/`OnOpened` как делегаты — это утечка памяти и риск вызова метода у уже уничтоженного объекта (объект скоупа уничтожен, но `EventBus` корневого скоупа продолжает жить). `IDisposable` + VContainer гарантируют, что `Dispose` будет вызван при уничтожении скоупа.

##### OnOpened сбрасывает флаг, а не конструктор

```csharp
private void OnModified(SceneModifiedEvent _) => _isDirty = true;
private void OnOpened(SceneOpenedEvent _)    => _isDirty = false;
```

> Флаг сбрасывается реакцией на `SceneOpenedEvent`, а не в конструкторе. Это важно: если `SceneDirtyTracker` живёт в сцен-скоупе, его конструктор вызывается один раз при загрузке сцены, до `SceneOpenedEvent`. Начальное значение `_isDirty = false` (дефолт `bool`) совпадает с желаемым — но логически сброс «по событию открытия» яснее выражает намерение: каждое открытие/создание сцены — чистая точка отсчёта.

##### Дискард параметра события

```csharp
private void OnModified(SceneModifiedEvent _) => _isDirty = true;
```

> `_` — дискард-параметр. `SceneModifiedEvent` — `struct` без полей (или с полями, которые здесь неважны). Синтаксис явно сообщает читателю: «содержимое события не используется, важен лишь факт его прихода». Это соответствует паттерну: `SceneDirtyTracker` не фильтрует по типу изменения — любое `SceneModifiedEvent` устанавливает флаг.

### К защите

> [!question] Вероятные вопросы
>
> **В:** Зачем отдельный класс для одного булевого флага?
> О: Потому что флаг «грязной» сцены — отдельная ответственность. Альтернатива — поле `bool isDirty` в `SceneGraph` или `AppStorage` — нарушила бы принцип единственной ответственности и создала бы прямую зависимость между подсистемами. `SceneDirtyTracker` — единственная точка, знающая об этом состоянии; остальные подсистемы только публикуют события, не зная о флаге.
>
> **В:** Почему подписка в `Start()`, а не в конструкторе?
> О: VContainer вызывает `IStartable.Start()` после построения всего графа зависимостей скоупа. Подписка в конструкторе означала бы получение событий до полной инициализации сцены — рецепт порядко-зависимых багов. `IStartable` — именно VContainer-контракт для «пост-инициализации».
>
> **В:** Что будет, если не вызвать `Unsubscribe` в `Dispose`?
> О: `EventBus` (корневого скоупа) сохранит делегаты `OnModified`/`OnOpened` как живые ссылки. Объект `SceneDirtyTracker` не будет собран GC, пока `EventBus` жив. Хуже: при следующем открытии сцены будет создан новый экземпляр, а старый продолжит получать события — двойная реакция на изменения.
>
> **В:** Как `SceneDirtyTracker` связан с сохранением?
> О: Никак напрямую. Флаг `IsDirty` читает навигационная логика (чтобы предупредить о несохранённых изменениях перед переходом). Сохранение выполняет [[SceneAutoSaver]], реагируя на `ModeExitingEvent` независимо от флага. `ClearDirty()` вызывается после ручного сохранения. Это разделение ответственностей: трекер только отслеживает, автосейвер только сохраняет.
