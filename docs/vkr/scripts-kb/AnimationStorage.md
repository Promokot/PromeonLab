> [!info] Назначение
> `AnimationStorage` — класс хранения анимационных данных: загрузка `animation.json` с неразрушающей обработкой несовместимой версии, отложенная запись с дебаунсом 200 мс. Относится к подсистеме **Animation**. Фрагмент загрузки — **Листинг 3.7**; фрагмент отложенной записи — **Листинг 3.10**; полный текст — **Приложение Б, Листинг Б.2**.

### Обзор

##### Роль и место

Регистрируется в сцен-скоупе `VrEditing`. Инъектируется в `AnimationAuthoring`, которая использует `LoadAsync` при открытии сцены и `RequestSave` после каждого изменения ключей. Реализует `IDisposable`: при уничтожении скоупа отменяет незавершённый дебаунсный таймер.

Ключевое архитектурное решение — **два режима работы с версиями схемы**: анимация не мигрируется аддитивно (в отличие от `SceneSerializer`), а при несовместимой версии возвращает пустой документ, не трогая файл на диске.

##### Ключевые методы

| Метод | Суть |
|---|---|
| `LoadAsync(sceneId, ct)` | Читает `animation.json`, проверяет версию, возвращает данные или пустой документ |
| `RequestSave(data, sceneId)` | Отменяет предыдущий дебаунсный таймер, запускает новый на 200 мс |
| `DebouncedSave(data, sceneId, ct)` | `Task.Delay` → `SaveAsync` (если не отменено) |
| `SaveAsync(data, sceneId, ct)` | `JsonUtility.ToJson` → `File.WriteAllTextAsync` |
| `Dispose()` | `_saveCts?.Cancel()` + `Dispose()` |

### Разбор кода

##### LoadAsync — строгая проверка версии

```csharp
if (loaded == null || loaded.schemaVersion < 2 || loaded.schemaVersion > 2)
{
    Debug.LogWarning($"AnimationStorage: '{path}' has unsupported schemaVersion="
        + $"{loaded?.schemaVersion ?? 0}. Opening empty; file left untouched.");
    return new SceneAnimationData();
}
```

> Условие `schemaVersion < 2 || schemaVersion > 2` — эквивалент `schemaVersion != 2`. Запись в таком виде делает намерение явным: «принимается только строго версия 2». Слева граница — защита от старых файлов, справа — от файлов из будущей версии приложения. В обоих случаях: возврат `new SceneAnimationData()` (пустой документ), файл не трогается. Это принципиальное отличие от [[SceneSerializer]], где v1 и v2 приводятся аддитивно.

##### LoadAsync — резервный Fps

```csharp
if (loaded.Fps <= 0)
    loaded.Fps = loaded.Containers.Count > 0 ? Mathf.Max(1, loaded.Containers[0].Fps) : 24;
```

> Защита от нулевого или отрицательного `Fps` (повреждённый файл или гипотетическая схема без общесценового `Fps`). Логика: берётся `Fps` первого контейнера, но не менее 1 (`Mathf.Max(1, ...)`); если контейнеров нет — дефолт 24. `Mathf.Max(1, ...)` важен: если и у контейнера `Fps == 0`, деление на него в `AnimationClock` дало бы `DivisionByZeroException` или `Infinity`.

##### LoadAsync — обработка исключений

```csharp
catch (Exception ex)
{
    Debug.LogError($"AnimationStorage: load failed '{path}': {ex.Message}");
    return new SceneAnimationData();
}
```

> Любая ошибка чтения/парсинга (битый файл, I/O ошибка, `OperationCanceledException` при отмене) перехватывается и возвращает пустой документ. Сцена открывается без анимации вместо краша. Ошибка логируется в журнал Unity — разработчик видит причину в консоли. `OperationCanceledException` тоже попадёт сюда — это допустимо, так как пустой документ при отмене безвреден.

##### RequestSave — дебаунс через отмену предыдущего токена

```csharp
public void RequestSave(SceneAnimationData data, string sceneId)
{
    _saveCts?.Cancel();
    _saveCts = new CancellationTokenSource();
    _ = DebouncedSave(data, sceneId, _saveCts.Token);
}
```

> `_saveCts?.Cancel()` отменяет предыдущий `DebouncedSave`, если он ещё ждёт в `Task.Delay`. Затем создаётся **новый** `CancellationTokenSource` и запускается новый таймер. Старый `CancellationTokenSource` при этом утекает (не `Dispose`-ится немедленно) — это допустимо, т.к. `Dispose()` класса вызовет итоговый `_saveCts?.Cancel()` + `_saveCts?.Dispose()`. `_ =` («discard») подавляет предупреждение компилятора «awaitable result ignored» — намеренно: вызывающий код не должен ждать записи.

##### DebouncedSave — поглощение TaskCanceledException

```csharp
private async Task DebouncedSave(SceneAnimationData data, string sceneId, CancellationToken ct)
{
    try
    {
        await Task.Delay(SAVE_DEBOUNCE_MS, ct);
        if (!ct.IsCancellationRequested) await SaveAsync(data, sceneId, ct);
    }
    catch (TaskCanceledException) { }
}
```

> `Task.Delay(200, ct)` прерывается отменой и бросает `TaskCanceledException`. Пустой `catch` — намеренное поглощение: отмена означает «пришёл новый `RequestSave`, старый дебаунс устарел». Ничего не делать — правильное поведение. Дополнительная проверка `!ct.IsCancellationRequested` перед `SaveAsync` — защита от гонки: между прохождением `Task.Delay` и вызовом `SaveAsync` токен может быть отменён конкурентным `RequestSave`.

##### Dispose — корректная финализация

```csharp
public void Dispose()
{
    _saveCts?.Cancel();
    _saveCts?.Dispose();
}
```

> При уничтожении скоупа (выход из VrEditing) VContainer вызывает `Dispose`. Сначала отмена ожидающего `DebouncedSave` — данные не записываются. Это сознательно: `SceneAutoSaver` к этому моменту уже сохранил граф сцены через `AppStorage.SaveSceneAsync`, а анимация в момент перехода должна была быть записана ещё при последней паузе пользователя (дебаунс 200 мс). Незаписанные изменения за последние 200 мс теряются — это принятый компромисс.

### К защите

> [!question] Вероятные вопросы
>
> **В:** Почему анимация не мигрируется аддитивно, как схема сцены?
> О: Для анимационных дорожек безопасного совместимого преобразования нет: состав ключей, формат трансформаций, привязка дорожек к костям — всё это может кардинально меняться между версиями. Автоматическое «исправление» рискует незаметно исказить или потерять записанные ключевые кадры. Отказ от чтения сохраняет данные физически: файл остаётся на диске и доступен для восстановления вручную.
>
> **В:** Что такое дебаунс и зачем он здесь?
> О: Дебаунс — приём, при котором фактическое действие откладывается до паузы в потоке событий. Каждая правка ключа вызывает `RequestSave`, который сбрасывает таймер на 200 мс. Пока пользователь активно правит, диск не затрагивается. Запись происходит в первую же паузу. Это превращает серию из сотен правок в одну дисковую операцию, снижая нагрузку на хранилище устройства.
>
> **В:** Что происходит с незаписанными изменениями при выходе из сцены?
> О: `Dispose()` отменяет ожидающий дебаунсный таймер. Изменения за последние 200 мс могут потеряться. Это принятый компромисс: пользователь ожидает, что выход из сцены — граница, и данные, записанные до паузы перед выходом, сохранены. На практике потеря не более 200 мс работы — приемлемый риск для данного приложения.
>
> **В:** Почему `_ = DebouncedSave(...)` вместо `await`?
> О: Вызов `RequestSave` синхронный: вызывающий код (`AnimationAuthoring`) не должен ждать записи и блокировать UI-поток. Дисcard-оператор `_ =` явно показывает, что Task намеренно «отпускается». Ошибки внутри `DebouncedSave` и `SaveAsync` обрабатываются внутри через `Debug.LogError` — они не пробросятся в вызывающий код.
