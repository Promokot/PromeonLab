> [!info] Назначение
> `SceneSerializer` — статический класс, выполняющий сериализацию `SceneData` в JSON и обратно, включая инлайн-миграцию устаревших версий схемы. Относится к подсистеме **StorageCore**. Полный текст приведён в **Листинге 3.6** раздела 3.1.3 ВКР.

### Обзор

##### Роль и место

`SceneSerializer` — единственное место, где происходит преобразование между `SceneData` и строкой JSON. Используется в [[AppStorage]] (`LoadSceneAsync` → `Deserialize`, `SaveSceneAsync` → `Serialize`). Класс `static` — у него нет состояния и нет зависимостей; не регистрируется в DI-контейнере. Миграция выполняется инлайн при десериализации: прочитанный объект дополняется до актуальной схемы прямо в методе `Deserialize`, без отдельного класса-мигратора. На диск файл попадает уже в актуальной версии при ближайшем сохранении.

##### Ключевые методы

| Метод | Суть |
|---|---|
| `Serialize(data)` | `JsonUtility.ToJson(data, prettyPrint: true)` — одна строка |
| `Deserialize(json)` | Разбирает JSON, проверяет `null`, выполняет цепочку миграций v1→v2→v3 |

### Разбор кода

##### Deserialize — защита от null и пустой строки

```csharp
public static SceneData Deserialize(string json)
{
    if (string.IsNullOrEmpty(json)) return null;
    var data = JsonUtility.FromJson<SceneData>(json);
    if (data == null) return null;
    ...
}
```

> Два отдельных `null`-теста. Первый: `string.IsNullOrEmpty(json)` — защита от вызова `FromJson` с `null` или пустой строкой, что бросило бы исключение. Второй: `data == null` — `JsonUtility.FromJson` вернёт `null` при синтаксически некорректном JSON (не-JSON строка, битый файл). Оба теста нужны: пустая строка проходит синтаксически, но `FromJson<SceneData>("")` вернёт `null`.

##### Миграция v1→v2 — цепочкой, не вложением

```csharp
if (data.SchemaVersion < 2)
{
    Debug.LogWarning($"SceneSerializer: migrating scene '{data.SceneId}' from v{data.SchemaVersion} to v2");
    data.SchemaVersion = 2;
    data.Nodes ??= new List<NodeData>();
}
if (data.SchemaVersion < 3)
{
    Debug.LogWarning($"SceneSerializer: migrating scene '{data.SceneId}' from v{data.SchemaVersion} to v3");
    data.SchemaVersion = 3;
    foreach (var n in data.Nodes) n.BonePoses ??= new List<BonePose>();
}
```

> Два последовательных `if`, а не `else if`. Файл версии 1 пройдёт оба блока: сначала дополнится до v2 (получит непустой `Nodes`), затем до v3 (каждой ноде инициализируется `BonePoses`). Файл v2 пройдёт только второй блок. Это аддитивная цепочка миграций: каждый шаг предполагает, что предыдущий уже выполнен. Добавление v4 потребует лишь третьего `if (data.SchemaVersion < 4)` — без изменения предыдущих шагов.

##### Null-coalescing при инициализации списков

```csharp
data.Nodes ??= new List<NodeData>();
// ...
foreach (var n in data.Nodes) n.BonePoses ??= new List<BonePose>();
```

> `??=` — оператор присваивания с нулевой проверкой (C# 8). `JsonUtility` оставляет список `null`, если в JSON поле отсутствует (для `List<T>` — в отличие от `string`, где возвращается пустая строка). `??=` заменяет `null` пустым списком, не трогая уже заполненные значения. Это важно для v2→v3 миграции: `data.Nodes` уже не `null` после первого шага, поэтому его `??=` не сработает — но `n.BonePoses` у каждой ноды старого файла всё равно `null`.

##### Serialize — prettyPrint

```csharp
public static string Serialize(SceneData data) =>
    JsonUtility.ToJson(data, prettyPrint: true);
```

> `prettyPrint: true` означает JSON с отступами. Это сознательный выбор: файлы на диске читаемы глазами, что упрощает отладку и ручную верификацию данных вне приложения (например, при тестировании или отчёте об ошибке). Плата — несколько большие файлы, что при сценах с десятками нод и костей несущественно.

### К защите

> [!question] Вероятные вопросы
>
> **В:** Почему миграция выполняется прямо в `Deserialize`, а не в отдельном классе?
> О: Миграция — неотъемлемая часть чтения данных. Разделение на «читать» и «мигрировать» породило бы риск забыть вызвать миграцию. Инлайн-подход гарантирует: любой код, получивший `SceneData` от `SceneSerializer.Deserialize`, уже работает с актуальной версией. CLAUDE.md явно предписывает: «migrations are inline at the deserialization boundary».
>
> **В:** Что произойдёт, если добавить v4 схемы?
> О: Добавляется третий блок `if (data.SchemaVersion < 4)` после существующих двух. Файлы v1 пройдут все три блока, v2 — два последних, v3 — только последний. Предыдущие блоки не меняются — это и есть аддитивность: каждый шаг дополняет данные, не ломая предыдущие.
>
> **В:** Почему схема сцены использует аддитивное приведение, а анимации — нет?
> О: Схема сцены развивалась аддитивно: новые версии добавляли поля без изменения смысла существующих, и дополнение значениями по умолчанию безопасно. Для анимационных дорожек совместимое преобразование между версиями не определено: автоматическое «исправление» рисковало бы незаметно исказить записанные ключевые кадры. Поэтому `AnimationStorage` при неподдерживаемой версии открывает пустой документ, не трогая файл.
>
> **В:** Почему используется `JsonUtility`, а не `Newtonsoft.Json` или `System.Text.Json`?
> О: `JsonUtility` входит в состав Unity и гарантированно работает на Quest без дополнительных зависимостей. Он сериализует `Vector3` и `Quaternion` «из коробки» как Unity-структуры. Ограничение — `public`-поля, нет поддержки полиморфизма — для текущей плоской структуры данных это несущественно.
