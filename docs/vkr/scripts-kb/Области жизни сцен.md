> [!info] Назначение
> Три внутрисценные области (`MainMenuSceneScope`, `VrEditingSceneScope`, `SandboxSceneScope`) задают состав объектов каждого режима. Каждая создаётся вместе со своей сценой и уничтожается при её выгрузке. Листинги 3.15 (MainMenu), 3.16 (VrEditing), Б.4 (Sandbox).

### Обзор

##### Роль и место

Каждому режиму соответствует своя сцена Unity и своя `LifetimeScope`. VContainer вкладывает внутрисценный контейнер в корневой [[RootLifetimeScope]]: сценовый объект может зависеть от корневого (например, `SceneAutoSaver` принимает `AppStorage` и `EventBus`), но не наоборот. Одновременно существует только одна внутрисценная область.

| Область | Содержимое | Особенности |
|---|---|---|
| `MainMenuSceneScope` | `SceneDirtyTracker`, `ScenePickerPanel`, `MainMenuPanel` | Минимальный состав; нет анимации, нет автосейва |
| `VrEditingSceneScope` | Всё из Sandbox + `AnimationClock`, `AnimationAuthoring`, `AnimationStorage`, `AnimationPlaybackSampler`, `SceneAutoSaver` | Полный режим с анимацией и автосохранением |
| `SandboxSceneScope` | `SceneGraph`, `SelectionManager`, `BoneEditMode`, `AssetSpawner`, `GizmoDriver`, … | Нет `SceneDirtyTracker`, нет автосейва, нет анимации |

##### Ключевые методы

Все три класса переопределяют только `Configure(IContainerBuilder)`. Никакой другой логики нет.

### Разбор кода

##### MainMenuSceneScope — RegisterComponentInHierarchy

```csharp
public class MainMenuSceneScope : LifetimeScope
{
    protected override void Configure(IContainerBuilder builder)
    {
        builder.Register<SceneDirtyTracker>(Lifetime.Scoped).AsImplementedInterfaces().AsSelf();
        builder.RegisterComponentInHierarchy<ScenePickerPanel>();
        builder.RegisterComponentInHierarchy<MainMenuPanel>();
    }
}
```

> `RegisterComponentInHierarchy<T>()` — VContainer сам делает `FindObjectOfType<T>()` по иерархии загруженной сцены и берёт найденный MonoBehaviour на учёт. Это легальный аналог `Find*` внутри `Configure`. `SceneDirtyTracker` зарегистрирован через `.AsImplementedInterfaces().AsSelf()` — он регистрируется одновременно как конкретный тип и как все интерфейсы, которые реализует. Это позволяет инжектировать его как по имени класса, так и по интерфейсу.
>
> Примечательно, что `SceneContextBinder` здесь **не регистрируется**: в MainMenu нет `SceneGraph` и нет сценовых сервисов, которые нужно отражать в [[SceneContext]]. Поэтому при входе в главное меню `ctx.HasScene == false` — это штатно.

##### VrEditingSceneScope — FindAnyObjectByType + RegisterBuildCallback

```csharp
var catcher = Object.FindAnyObjectByType<EmptySpaceClickDeselector>(FindObjectsInactive.Include);
if (catcher != null)
    builder.RegisterBuildCallback(c => c.Inject(catcher));
```

> Паттерн: `FindAnyObjectByType` находит MonoBehaviour в иерархии сцены. По соглашениям проекта это **единственное легальное место** для такого поиска — внутри `Configure` или `RegisterBuildCallback`. Затем `RegisterBuildCallback` откладывает инжект до момента, когда контейнер полностью построен (все конструкторы отработали). `c.Inject(catcher)` заполняет `[Inject]`-аннотированные поля/методы MonoBehaviour, который VContainer не создавал. Проверка `if (catcher != null)` — защита на случай, если объект не размещён в сцене.

##### VrEditingSceneScope — порядок регистрации анимационных Entry Points

```csharp
builder.RegisterEntryPoint<AnimationClock>(Lifetime.Scoped).AsSelf();
builder.Register<AnimationStorage>(Lifetime.Scoped).AsImplementedInterfaces().AsSelf();
builder.RegisterEntryPoint<AnimationPlaybackSampler>(Lifetime.Scoped).AsSelf();
builder.RegisterEntryPoint<AnimationAuthoring>(Lifetime.Scoped).AsSelf();
```

> `AnimationClock` и `AnimationPlaybackSampler` зарегистрированы как `RegisterEntryPoint` — VContainer вызовет их `ITickable.Tick()` каждый кадр. `AnimationStorage` — пассивный `Register`: он только читает/пишет файлы по запросу, ему покадровой работы не нужно. `AnimationAuthoring` — `RegisterEntryPoint` с `IStartable`: при старте области он загружает `animation.json` через `AnimationStorage`.

##### VrEditingSceneScope — финальный RegisterBuildCallback для роутера

```csharp
builder.RegisterBuildCallback(c =>
{
    var router = c.Resolve<PanelRegionRouter>();

    foreach (var rm in Object.FindObjectsByType<RegionMember>(FindObjectsInactive.Include, FindObjectsSortMode.None))
    {
        c.Inject(rm);
        router.RegisterModule(rm.ModuleId, rm);
    }

    router.ApplyMode(c.Resolve<ModeOrchestrator>().CurrentMode);
});
```

> `FindObjectsByType` (множественное число) — перебирает всех `RegionMember` в иерархии сцены, в том числе неактивных. Каждый инжектируется (`c.Inject`) и регистрируется в роутере. Вызов `router.ApplyMode(...)` в конце перестраивает видимость панелей для текущего режима — вызывается **один раз**, сразу после построения области.

##### SandboxSceneScope — что отсутствует и почему

```csharp
// Нет в SandboxSceneScope:
// builder.Register<SceneDirtyTracker>(...)
// builder.Register<SceneAutoSaver>(...)
// builder.RegisterEntryPoint<AnimationClock>(...)
// builder.RegisterEntryPoint<AnimationAuthoring>(...)
// builder.Register<AnimationStorage>(...)
// builder.RegisterEntryPoint<AnimationPlaybackSampler>(...)
```

> Sandbox — временное пространство для эксперимента, без сохранения и без анимации. Отсутствие `SceneDirtyTracker` означает, что сцена никогда не помечается «грязной» — нет смысла, если данные не сохраняются. Отсутствие `SceneAutoSaver` — не будет автосохранения при `ModeExitingEvent`. Отсутствие анимационных классов — `SceneContextBinder.Resolve<AnimationAuthoring>()` вернёт `null`, и `ctx.Authoring == null` при `ctx.HasScene == true`.

### К защите

> [!question] Вероятные вопросы
>
> **В:** Почему `FindAnyObjectByType` допустим в `Configure`, хотя в остальном коде запрещён?
> **О:** По соглашениям проекта, `FindObjectOfType` и аналоги запрещены в геймплейном коде, потому что создают неявные зависимости и мешают тестируемости. Исключение — DI-bootstrap-шим: `Configure` и `RegisterBuildCallback` в `LifetimeScope` — это единственная точка, где MonoBehaviour нужно найти в сцене и «отдать» контейнеру. После этого все остальные объекты получают зависимость через конструктор.
>
> **В:** Почему Sandbox не регистрирует `SceneDirtyTracker`?
> **О:** `SceneDirtyTracker` отслеживает изменения, чтобы знать, что сцену нужно сохранить. В Sandbox сохранения нет по замыслу — это временное пространство для эксперимента. Регистрировать трекер, который никогда не будет использован (нет `SceneAutoSaver`, нет сохранения), не нужно.
>
> **В:** В чём разница между `RegisterComponentInHierarchy` и паттерном `FindAnyObjectByType + RegisterBuildCallback`?
> **О:** `RegisterComponentInHierarchy<T>()` — VContainer сам находит компонент и регистрирует его для инжекции в других объектов (как поставщик). Паттерн с `FindAnyObjectByType + RegisterBuildCallback(c => c.Inject(...))` — объект **получает** зависимости из контейнера, но сам не обязательно поставляется другим. Часто оба подхода комбинируют: `RegisterInstance(obj)` чтобы зарегистрировать как поставщика, и `RegisterBuildCallback(c => c.Inject(obj))` чтобы наполнить его зависимостями.
>
> **В:** Что происходит со сценовой областью при смене режима?
> **О:** `LoadSceneMode.Single` выгружает все объекты текущей сцены. VContainer отслеживает это и вызывает `IDisposable.Dispose()` на все объекты сценовой области, в том числе на `SceneContextBinder` (который очищает [[SceneContext]]) и `SceneAutoSaver` (который сохраняет). Новая сцена загружается, её `LifetimeScope` строит новую область и вкладывает её в корневую.
>
> **В:** Почему `SceneContextBinder` зарегистрирован в VrEditing и Sandbox, но не в MainMenu?
> **О:** В MainMenu нет сценовых сервисов, которые нужно отражать в `SceneContext` — нет `SceneGraph`, нет анимации. `SceneContext` должен быть пустым (`HasScene = false`) в главном меню — это корректное состояние. Регистрировать `SceneContextBinder` только чтобы он вызвал `Bind` с четырьмя `null`-значениями смысла нет.
