> [!info] Назначение
> `AssetEntityRecipe` и `AssetRef` — структуры данных, описывающие запись библиотеки ассетов и ссылку на неё из сцены. Относятся к подсистеме **AssetBrowser**. Поля обоих типов приведены в **Листинге 3.3** раздела 3.1.2 ВКР.

### Обзор

##### Роль и место

`AssetEntityRecipe` реализует паттерн **build-once / restore-many**: ассет обрабатывается один раз при импорте (вычисляются габариты коллайдера, тип взаимодействия, описание скелета), результат сохраняется в рецепте. При каждом размещении объекта в сцене рецепт применяется в готовом виде через `IAssetEntityBuilder` → `AssetEntityBuilderRegistry.RestoreAsync`. Повторного разбора модели не требуется, представление объекта не меняется от запуска к запуску.

`AssetRef` — минимальная ссылка: `Source` (откуда взят ассет) + `AssetId` (идентификатор записи в соответствующей библиотеке). Хранится в `NodeData` и указывает `AssetSpawner`, где искать рецепт при загрузке сцены.

##### Ключевые методы

`AssetEntityRecipe`:
- `HasRig` — вычисляемое свойство, проверяет наличие скелета через счётчик костей (не через `null`).

`AssetRef` — только данные; `ToString()` возвращает `"Source/AssetId"` для логов.

### Разбор кода

##### AssetEntityRecipe — защита HasRig от null-ловушки JsonUtility

```csharp
// Canonical "this recipe describes a skeleton" check. JsonUtility cannot persist a null nested
// object, so `rig` may come back as an empty object – guard on the bone count, never on null.
public bool HasRig => rig != null && rig.Bones != null && rig.Bones.Count > 0;
```

> `JsonUtility` не сериализует `null` для вложенных классов, помеченных `[Serializable]`: при десериализации поле `rig` будет ненулевым объектом с пустыми коллекциями, даже если ассет вообще не имел скелета. Проверка `rig != null` необходима для случая, когда объект создаётся в коде без десериализации, но недостаточна после загрузки с диска. Поэтому канонический тест — `rig.Bones != null && rig.Bones.Count > 0`. Если написать просто `rig != null`, у всех не-rig ассетов после перезапуска ложно сработает ветка скелета.

##### AssetEntityRecipe — spawnOffset vs. сохранённая позиция

```csharp
// Initial placement: world-space offset applied ONCE at fresh spawn (reload uses the saved pos).
public Vector3 spawnOffset;
```

> `spawnOffset` применяется только при **первом** размещении ассета (fresh spawn). При перезагрузке сцены объект восстанавливается по `NodeData.Position`, которая уже учитывает смещение. Это разделение важно: без него каждый перезапуск сцены сдвигал бы объект снова и снова.

##### AssetRef — struct, а не class

```csharp
[Serializable]
public struct AssetRef
{
    public AssetSource Source;
    public string      AssetId;
    ...
}
```

> `AssetRef` — `struct` без поведения, хранится по значению внутри `NodeData`. `JsonUtility` корректно сериализует `struct`, и значение-по-умолчанию (`Source = 0`, `AssetId = null`) не требует явной инициализации. Сравнение через `==` не определено — используется только для чтения полей, что достаточно для роли ссылки.

##### schemaVersion у рецепта

```csharp
public int schemaVersion = 1;
```

> Текущая версия рецепта — 1, то есть формат ни разу не менялся с момента введения. Поле присутствует по соглашению: все сериализуемые структуры обязаны его содержать, чтобы будущая миграция была возможна без изменения формата существующих файлов.

### К защите

> [!question] Вероятные вопросы
>
> **В:** Что такое «build-once / restore-many» и зачем он нужен?
> О: При импорте ассет обрабатывается один раз: вычисляются коллайдеры, слои взаимодействия, описание скелета — всё записывается в рецепт. При каждом открытии сцены или размещении объекта рецепт применяется напрямую без повторного разбора модели. Это гарантирует, что представление объекта не меняется между запусками, и исключает затратный разбор файлов во время VR-сессии.
>
> **В:** Почему `HasRig` проверяет `Bones.Count`, а не просто `rig != null`?
> О: Потому что `JsonUtility` не сохраняет `null` для вложенных `[Serializable]`-классов: после десериализации `rig` будет ненулевым объектом даже у не-скелетных ассетов. Единственный надёжный признак наличия скелета — непустой список костей.
>
> **В:** Зачем `AssetRef` — `struct`, а не `class`?
> О: Ссылка на ассет — чистые данные без поведения, два поля. Тип-значение не требует отдельной аллокации, не будет `null` при десериализации `JsonUtility`, и семантика копирования по значению точно соответствует роли «адрес записи в библиотеке».
>
> **В:** Что означает `AssetSource` в `AssetRef`?
> О: Перечисление из трёх значений: `Builtin` (встроенные ассеты, поставляются с приложением), `Imported` (импортированные пользователем glTF/GLB/изображения), `Saved` (сохранённые из сцены, пока не реализованы). По `Source` `AssetSpawner` выбирает нужную библиотеку для поиска рецепта.
