# Контекст ВКР — VR-приложение для 3D-анимации

> **Примечание:** AR-составляющая (пространственная разметка, ArUco-маркеры, привязка к физической локации) была удалена из проекта 2026-05-21. Раздел "Новизна" ниже отмечен как требующий пересмотра — автор должен сформулировать новый тезис новизны без AR.

## Тема, цель и задачи

**Тема:** Разработка приложения для создания объектной и скелетной анимации трёхмерных моделей с применением технологий VR.

**Цель:** Разработать приложение для создания и редактирования скелетной и объектной анимации в иммерсивной VR-среде с последующим экспортом для постобработки во внешних редакторах.

**Задачи:**
1. Провести анализ существующих VR-решений для 3D-анимации, определить преимущества и недостатки.
2. Исследовать методы и подходы к созданию анимации в VR-среде.
3. Спланировать архитектуру функциональных модулей приложения и реализовать их как цельный продукт.
4. Обеспечить вывод анимированной сцены в стандартный или внутренний формат с сохранением структурных данных и анимационных ключей.

## Глава 1 — Аналитический раздел

### 1.1 Анализ существующих VR-решений

Проанализированы три приложения: **Quill**, **Tvori**, **AnimVR**. Критерии: платформенная доступность, условия распространения, таймлайн/ключевые кадры, объектная анимация, риггинг и скелетная анимация, импорт/экспорт.

**Quill** (Smoothstep) — рисование в VR с анимационными возможностями. Таймлайн есть, покадровый подход + «анимационная кисть» + марионеточная анимация. Риггинг отсутствует. Импорт FBX/OBJ (статика), экспорт USD/Alembic/FBX. Частично автономен.

**Tvori** — среда для аниматиков и превизуализаций. Таймлайн есть. Частичный скелетный позинг через IK, импорт моделей со скелетом. Экспорт нестабильный. Поддерживает автономную работу.

**AnimVR** (NVRMIND) — покадровая рисованная анимация. Развитый таймлайн. Риггинг отсутствует. Широкий экспорт (FBX/OBJ/glTF/USD/Alembic/video/.exe). Только PC VR.

**Вывод:** Скелетная анимация в существующих VR-приложениях либо отсутствует, либо ограничена; это обосновывает разработку собственного решения с акцентом на VR-нативное управление ригами, позами и ограничителями.

### 1.2 Методы анимации в VR

Рассмотрена адаптация классических анимационных подходов к VR-среде:

- **Таймлайн** — сохраняет роль, но требует адаптации под пространственные жесты; плоский интерфейс неудобен.
- **Объектная анимация** — естественна для VR (захват и перемещение объектов).
- **Риггинг и скелетная анимация** — методы (FK/IK, ограничители, весовые группы) остаются теми же; однако полная настройка рига выполняется во внешнем редакторе, VR-среда берёт на себя только позинг и запись ключей.
- **Интерполяция** — для VR достаточны линейная и ступенчатая; сплайновая настройка откладывается на постобработку в десктопном редакторе.

### 1.3 Технологии VR-трекинга

Используется **inside-out tracking** (Meta Quest 2/3): IMU (акселерометр + гироскоп) + визуальная одометрия → 6DoF без внешних камер. Для сглаживания траектории — экспоненциальный фильтр или фильтр Калмана.

### 1.4 Требования и стек

**Функциональные требования** (5 областей):
1. Загрузка и хранение ресурсов (статические меши, скелетные персонажи, материалы/текстуры; библиотека ассетов).
2. Формирование сцены (создание/загрузка пространств, размещение/трансформация объектов, иерархия родитель–потомок, аутлайнер).
3. Скелетные структуры и ограничители (управление костями, прокси-контроллеры, constraints, IK).
4. Анимация (запись ключевых кадров, таймлайн, редактирование ключей, линейная и ступенчатая интерполяция).
5. Экспорт (стандартные форматы OBJ/FBX; собственный формат проекта).

**Технологический стек:**
- **Unity** + **C#** — основная среда разработки
- **OpenXR** — кроссплатформенный VR-слой (без привязки к Meta SDK)
- **Git / GitHub** — контроль версий
- **Blender 3D + Python API** — целевой редактор постобработки; при необходимости — собственный аддон-импортер

## Глава 2 — Специальный раздел (планирование подсистем)

> **Важно:** Глава 2 содержит **не готовую архитектуру реализации**, а **проектирование и планирование** функциональных подсистем — их состав, зоны ответственности и схемы взаимодействия. Конкретные архитектурные решения (паттерны, DI, event-шины, bootstrapping) описываются и детализируются отдельно, вне курсовой.

Система декомпозирована на **5 взаимосвязанных подсистем**, организованных как направленный конвейер: импорт → сцена → риг/ограничители → анимация → экспорт. Базовый уровень взаимодействия пользователя (перемещение, работа с контроллерами) является сквозным для всех подсистем.

**Режимы перемещения:** физическое перемещение в пределах зоны трекинга + виртуальное (плавное по стикам, телепортация, поворот камеры).

### 2.1 Подсистема управления файловыми ресурсами
*(соответствует ProjectStorage + AssetCatalog)*

Входной и связующий компонент. Импорт внешних ассетов (статические меши, скелетные персонажи с развесовкой, материалы/текстуры) с нормализацией во внутренний формат. Централизованный хаб данных: сцены, ключевые позы, анимационные последовательности, параметры ригов и констрейнтов. Часть данных сериализуется автоматически. Позы сохраняются как переиспользуемые ассеты. Пользовательский интерфейс — виртуальная библиотека ассетов (файловый проводник).

### 2.2 Подсистема формирования и редактирования сцен
*(соответствует SceneComposition)*

Организация рабочего 3D-пространства. Сцена главного меню — точка входа для управления проектами. Аутлайнер — список объектов, выбор, удаление, иерархия родитель–потомок. Инстанцирование ассетов из библиотеки с последующими трансформациями (перемещение, вращение, масштабирование).

### 2.3 Подсистема ограничителей и скелетных структур
*(соответствует RigBuilder)*

Создание, визуализация и настройка манипуляторов для объектов и скелетных мешей. Визуализация костей — пирамидальный меш от родителя к дочернему элементу (вместо пустых объектов Unity). По умолчанию для костей разрешено только вращение (запрет перемещения для сохранения иерархии). Ручное добавление ограничителей: блокировка осей, диапазоны трансформаций, IK-цепочки с заданной глубиной. Прокси-контроллеры с произвольными мешами из библиотеки. Риги сохраняются как префабы-шаблоны в библиотеке ассетов. Настройка скелета (весовые группы, создание иерархии) выполняется во внешнем редакторе до импорта.

### 2.4 Подсистема анимации
*(соответствует AnimationAuthoring + AnimationPlayback)*

Запись, хранение и воспроизведение временных изменений состояния объектов и скелетных структур. Анимационные контейнеры (клипы) — привязаны к конкретным объектам/костям, хранят ключевые кадры независимо. Ключевой кадр фиксирует позицию/вращение/масштаб/свойства объекта или состояние костей скелета. Позы сохраняются как отдельные переиспользуемые сущности (применение к совместимым ригам). Таймлайн — линейная структура с операциями добавления/удаления/перемещения/перезаписи ключей. Интерполяция: линейная и ступенчатая (на выбор для каждого ключа); сплайны — на этапе постобработки вне VR.

### 2.5 Подсистема экспорта данных
*(соответствует ExportPipeline)*

Формирование итогового представления сцены для внешних редакторов. Обратный импорт не предусмотрен. Сбор данных: структура сцены, трансформации объектов, ключевые кадры, позы → нормализация в единую экспортную модель.

Состав пакета: геометрия сцены, анимационные данные (ключевые кадры, последовательности поз, интерполяция), структурная информация (иерархия объектов).

Форматы вывода:
- **Стандартные** (FBX, OBJ и подобные) — для передачи в внешние редакторы/движки.
- **Собственный формат проекта** — для случаев, когда стандартные форматы не сохраняют полноту данных (высокая плотность ключей, нестандартная иерархия ригов). Допускается создание плагина-импортера для Blender.
